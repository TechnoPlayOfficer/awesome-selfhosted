1. Resumen Ejecutivo
1.1 Propósito del Documento
Este PRD define los requerimientos funcionales, lineamientos de implementación y el plan de trabajo del Admin Dashboard de ACCET: el panel de operaciones internas que centraliza todos los flujos del negocio. Es el tercer PRD de la serie Dual-Ledger y consume los datos expuestos por Formance Ledger (v1.0) y Hedera HTS (v1.0).
El Admin Dashboard es el sistema nervioso central de AgoraLink S.A.S.: desde aquí el CEO toma decisiones estratégicas, el CFO aprueba distribuciones, el CTO gestiona la infraestructura blockchain, el equipo de Ops onboardea propietarios e inversores, y Compliance genera los reportes regulatorios para la SFC y la DIAN.
1.2 Problema que Resuelve
Sin un Admin Dashboard integrado, el equipo ACCET opera con cinco fricciones críticas que escalan con cada activo nuevo:
El pipeline de solicitudes de propietarios se gestiona en WhatsApp y hojas de cálculo — sin estado centralizado, los activos se pierden o demoran por falta de visibilidad.
Las aprobaciones del comité de activos (CEO + CFO) requieren reuniones presenciales porque no existe un flujo digital de firma y registro auditado.
El CFO calcula y ejecuta distribuciones manualmente contra múltiples sistemas — el panel centralizado con Formance + Forwarder + Hedera es el único camino hacia la automatización sin errores.
Los reportes regulatorios (SFC Sandbox, medios magnéticos DIAN) se construyen manualmente — sin extracción automática desde los sistemas, el riesgo de error es alto y el tiempo de preparación excesivo.
No existe un audit trail centralizado de las acciones críticas del sistema — condición eliminatoria para el due diligence de inversores institucionales y para la certificación ISO 9001.

2. Arquitectura General del Admin Dashboard
2.1 Sistema de Roles y Acceso
El Admin Dashboard implementa RBAC (Role-Based Access Control) con 7 roles internos. Cada rol tiene acceso exclusivamente a los módulos y acciones que requiere su función. El principio de mínimo privilegio aplica en cada pantalla y cada botón.

2.3 Principios de Diseño UI/UX del Admin Dashboard
Contexto siempre cargado: cuando un usuario abre un ticket, el expediente del propietario o inversor está pre-cargado. No se pide repetir información al usuario externo ni al agente interno.
Acciones destructivas requieren confirmación doble: aprobar un activo, ejecutar una distribución, quemar tokens o pausar contratos siempre presentan un modal de confirmación con resumen del impacto antes de ejecutar.
Estado visible en todo momento: el badge de entorno activo (PRODUCCIÓN / STAGING), el score de isomorfismo Formance-Hedera y el estado de Hedera mainnet son visibles en la barra superior sin importar en qué módulo esté el usuario.
Roles visualizados en color: cada rol tiene un color de acento en su badge de sesión. El CEO ve un badge dorado, el CFO azul, el CTO verde oscuro. Esto refuerza visualmente quién puede hacer qué en cada pantalla.
Audit trail no es un módulo aparte: cada acción crítica ejecutada desde el dashboard muestra un mini-log in-line (usuario, fecha, acción) directamente en la pantalla donde se ejecutó.

4. Fase 1 — Fundamentos: Auth, Roles y Audit Trail
4.1 Lineamientos — Fase 1
4.1.1 Sistema de Autenticación y 2FA (US-A-36)
El sistema de autenticación es la primera pantalla que cualquier usuario del equipo ve. Debe ser seguro, rápido y reforzar desde el inicio la cultura de seguridad de ACCET.
Login con email + contraseña. 2FA obligatorio en el primer login de cualquier usuario nuevo. Métodos permitidos: TOTP (Google Authenticator / Authy) y SMS OTP.
2FA adicional requerido en acciones críticas durante la sesión, aunque el usuario ya se haya autenticado: ejecutar distribución, aprobar activo, cambiar cuenta bancaria, emitir tokens, pausar contratos.
Sesión con TTL activo de 8 horas para roles operativos (Ops, Soporte). TTL de 4 horas para roles con acceso a acciones críticas (CEO, CFO, CTO, Compliance). Al expirar, la sesión se cierra y requiere re-autenticación.
Panel de sesiones activas: cada usuario puede ver todos sus dispositivos con sesión activa. Botón &apos;Cerrar todas las sesiones&apos; de emergencia disponible desde el perfil.
Alerta por email al detectar login desde IP o país nuevo. Log de todos los logins con timestamp, IP y dispositivo almacenado durante 90 días.
4.1.2 Gestión de Usuarios Internos (US-A-36)
Creación de usuario: email + rol + 2FA forzado en primer login. Solo el CEO y el CTO pueden crear usuarios con roles CEO, CFO, CTO o Compliance.
Roles disponibles: CEO · CFO · CTO · Ops · Compliance · Legal · Soporte. Un usuario puede tener un único rol principal. Los permisos son los definidos en la Sección 2.1.
Desactivación inmediata de usuario sin borrado: el historial de acciones del usuario desactivado queda preservado para auditoría. La desactivación requiere causa registrada en el audit trail.
Log de accesos por usuario: última sesión, IP, dispositivo, acciones ejecutadas durante la sesión. Retención de 90 días online + 1 año en archivo.
4.1.3 Audit Trail Inmutable (US-A-31)
Principio arquitectónico: El audit trail NO es un módulo opcional que se agrega al final. Es la columna vertebral del sistema. Cada acción crítica escribe al audit trail antes de confirmar su ejecución. Si el audit trail falla, la acción no se ejecuta.
Cada entrada del audit trail incluye: usuario · rol · acción · objeto afectado (activo_id, investor_id, etc.) · timestamp UTC · hash SHA-256 del registro · hash encadenado al registro anterior (blockchain interno).
Acciones on-chain (emisión, distribución, burn, whitelist): el audit trail incluye también el TX ID de Hedera como referencia cruzada permanente.
El audit trail es de solo lectura para todos los roles, incluido el CTO. Ningún usuario puede editar o eliminar una entrada. El motor de audit trail opera con una base de datos separada de append-only.
Exportable como PDF firmado digitalmente para entrega a auditor externo o regulador. El hash del export se registra en el mismo audit trail.
Filtros: por usuario · rol · tipo de acción · objeto afectado · rango de fechas · estado (exitoso/fallido).
4.1.4 Layout Base y Navegación
Barra superior permanente: logo ACCET + badge de rol del usuario activo (con color por rol) + badge de entorno activo (PRODUCCIÓN verde / STAGING amarillo) + score de isomorfismo Formance-Hedera + notificaciones.
Sidebar de navegación colapsable con los módulos activos para el rol del usuario. Los módulos inaccesibles para su rol no aparecen en el sidebar (no se muestran como disabled — directamente no existen para ese rol).
Centro de alertas accesible desde cualquier pantalla: ícono de campana con contador de alertas activas no resueltas. Click abre el panel lateral de alertas sin salir del módulo actual.

5. Fase 2 — Módulo de Onboarding de Activos y Propietarios
5.1 Lineamientos — Fase 2
5.1.1 Bandeja de Solicitudes (US-A-01)
La bandeja de solicitudes es la pantalla principal del equipo de Ops. Centraliza todas las solicitudes entrantes de propietarios y las mueve a través del pipeline de aprobación sin herramientas externas.
Tabla principal: ID solicitud · nombre propietario · ciudad · tipo inmueble · avalúo estimado · estado actual · tiempo en cola · analista asignado.
Estados del pipeline: Nueva → KYC Pendiente → Due Diligence → Legal Review → Aprobada → Rechazada → Emitida. Cada transición de estado se registra en el audit trail.
Alerta visual automática (fila destacada en rojo) si una solicitud lleva más de 48 horas en el mismo estado sin movimiento.
Filtros: por estado · por tipo de activo · por analista asignado · por fecha de ingreso. Ordenable por tiempo en cola para priorizar las más antiguas.
Asignación de solicitud a analista: dropdown con los usuarios del rol Ops disponibles. La asignación se registra en el audit trail con el usuario que la hizo.
5.1.2 Revisión Manual de KYC (US-A-02)
Panel KYC unificado: foto cédula frontal y trasera, selfie, resultado automático de Jumio/Onfido con score de confianza, y resultados de screening OFAC + SARLAFT Colombia + UN Consolidated con detalle por lista y timestamp.
Botones de acción: Aprobar / Rechazar / Solicitar re-envío de documento. Cada acción requiere campo de comentario obligatorio que se guarda en el expediente y en el audit trail.
Al aprobar KYC: el sistema dispara automáticamente la instrucción de alta en la whitelist ERC-3643 al Forwarder Ledger (insumo para el PRD del Forwarder).
Al rechazar KYC: se genera automáticamente la comunicación al propietario con la plantilla de rechazo correspondiente y los pasos de corrección.
5.1.3 Due Diligence y Checklist de Activo (US-A-03)
Checklist visual: escritura pública, certificado de libertad y tradición, paz y salvo predial, seguro vigente. Cada ítem con estado Cargado / Pendiente / Vencido y fecha de vigencia donde aplica.
Visor de PDF integrado: el analista puede ver cada documento sin salir de la pantalla. Click en el ícono de ojo abre el PDF en un panel lateral. No se requiere descarga para revisar.
Campo de notas por ítem: el analista puede dejar comentarios vinculados a cada documento. Las notas son visibles para todos los roles con acceso al expediente.
El botón &apos;Solicitar avalúo Mundo Hogar&apos; se habilita solo cuando el checklist está al 100%. Al hacer click, se genera automáticamente el formulario de solicitud con los datos del inmueble pre-llenados (US-A-04).
5.1.4 Aprobación del Comité y Firma Digital (US-A-06)
La aprobación del comité de activos es la acción más crítica del pipeline. Requiere la firma digital del CEO o CFO y queda registrada de forma permanente e inmutable.
Resumen ejecutivo del activo: nombre, avalúo, LTV, monto a emitir, rendimiento al propietario, fee ACCET proyectado, perfil de riesgo.
Semáforo de riesgo por dominio del audit: D1 Legal · D2 Técnico · D3 Valoración · D5 Riesgo — verde/amarillo/rojo. El CEO puede ver de un vistazo qué dominios tienen alertas antes de aprobar.
La aprobación requiere: firma digital del aprobador (PIN de 6 dígitos + confirmación 2FA) + registro en el audit trail con hash del expediente completo en ese momento.
Al aprobar: el sistema crea automáticamente las cuentas virtuales del activo en Formance (insumo Formance PRD US-L-03) y habilita el módulo de emisión para ese activo.
5.1.5 Comunicación de Rechazo (US-A-07)
Biblioteca de plantillas de rechazo por categoría: documentación incompleta, LTV excedido, falla KYC, inmueble no elegible. Cada plantilla es editable con texto libre antes de enviar.
Envío simultáneo por email + WhatsApp + push en la app del propietario. El envío se registra en el audit trail con el canal, la plantilla usada y el usuario que la envió.
La solicitud rechazada queda en estado &apos;Rechazada — Subsanable&apos; con 30 días para corrección. Al vencer el plazo, se archiva automáticamente.

6. Fase 3 — Módulo de Emisión On-chain y Cartera
6.1 Lineamientos — Fase 3
6.1.1 Configuración del Token y Despliegue ERC-3643 (US-A-08, US-A-09)
Este módulo es el puente entre el pipeline de aprobación de activos (Fase 2) y la capa Hedera. Una vez aprobado el activo, el CTO configura los parámetros del token y ordena el despliegue.
Formulario de configuración del token: nombre (ej. ACCET-NdS-001), símbolo, supply total, precio por token en USD, rendimiento E.A. al propietario, fee ACCET (3%), plazo en días, fechas de distribución. Todos los campos se pre-llenan desde el expediente del activo para eliminar re-ingreso manual.
Vista previa del contrato ERC-3643 con los parámetros ingresados antes de confirmar. El CTO puede revisar el bytecode resultante y el hash antes de ordenar el despliegue.
El botón &apos;Desplegar en Hedera mainnet&apos; es una acción destructiva: requiere confirmación doble (modal con resumen + 2FA del CTO). Está físicamente desactivado hasta que el tracker de auditoría del contrato tiene estado Aprobado (insumo Hedera PRD Sección 5).
Post-despliegue: el panel muestra inmediatamente el Token ID de Hedera, el TX ID de creación, el supply emitido y el link a hashscan.io. El estado del activo cambia automáticamente a &apos;Emitido · Abierto a suscripción&apos;.
Panel de timelock activo: cuenta regresiva visible para upgrades propuestos, con opción de cancelar para el quórum del directorio.
6.1.2 Monitor de Suscripción en Tiempo Real (US-A-11)
KPIs en vivo actualizados por WebSocket: tokens suscritos / supply total · porcentaje completado · capital recaudado en USD · número de inversores activos.
Gráfico de velocidad de suscripción: tokens suscritos por día desde la apertura. Permite al equipo detectar si la suscripción está estancada y tomar acción.
Lista de últimas suscripciones: hora, inversor (anonimizado para Ops, con nombre completo para Compliance), monto, método de pago (PSE/USDC/wire).
Botón &apos;Cerrar suscripción anticipada&apos;: disponible en cualquier momento con confirmación de doble factor. Al cerrar, el estado del activo cambia a &apos;Cerrado · Suscripción completada&apos; y se bloquean nuevas suscripciones.
6.1.3 Confirmación de Pagos y Bandeja de Emisión (US-A-12)
Bandeja unificada de pagos recibidos: inversor · monto · método · estado (pendiente/confirmado/fallido) · tiempo transcurrido desde el pago.
PSE y USDC: confirmación automática al recibir la señal del pasarela de pagos o del Forwarder. La emisión de tokens se dispara automáticamente al confirmar.
Wire: confirmación manual por el equipo de Ops con botón &apos;Marcar como recibido&apos; y campo de referencia del wire obligatorio. La confirmación registra en audit trail quién la ejecutó y con qué referencia.
Alerta si un pago PSE o USDC no se confirma en 15 minutos (posible falla técnica): escalamiento automático al CTO con detalle de la TX fallida.
6.1.4 Dashboard Ejecutivo de Cartera (US-A-15)
KPIs ejecutivos: AUM total USD · número de activos activos · número de inversores activos · rendimiento promedio E.A. de la cartera.
Ingresos ACCET: fees acumulados del período vs proyectado del modelo financiero MVP NdS 2026. Desviación porcentual visible para el CFO.
Mapa de estado de cartera: semáforo por activo (verde = sano, amarillo = atención, rojo = crítico). Click en cualquier activo abre el expediente completo.
Exportable como PDF ejecutivo para presentación al directorio en un click desde el panel.
6.1.5 Monitor LTV Oracle Chainlink (US-A-16)
Tabla LTV en tiempo real consumida del Mirror Node vía Chainlink: activo · avalúo actual · LTV actual · variación % vs emisión · estado semáforo.
Alerta automática al analista de riesgo si LTV ≥ 65% (amarillo) y al CEO si LTV ≥ 70% (rojo). Las alertas aparecen en el centro de alertas y se envían por email + push.
Histórico de LTV por activo: gráfico de línea de los últimos 12 meses para detectar tendencias de depreciación del colateral.
6.1.6 Whitelist ERC-3643 y Pausa de Emergencia (US-A-10, US-A-14)
Panel de whitelist: lista de inversores habilitados con AccountID Hedera, fecha de alta, TX ID de la operación on-chain y estado (activo/suspendido). La alta se ejecuta automáticamente al aprobar KYC desde la Fase 2.
Botón de pausa de emergencia: visible solo para CTO y CEO. Al presionarlo, aparece un modal que requiere la razón + 2FA del iniciador + aprobación del segundo firmante (si CTO inicia, CEO aprueba y viceversa). La pausa se ejecuta en el Forwarder hacia Hedera en menos de 60 segundos.

7. Fase 4 — Distribuciones, Compliance y Reportes
7.1 Lineamientos — Fase 4
7.1.1 Calendario Global de Distribuciones (US-A-22)
Vista calendario mensual con todos los eventos de distribución de todos los activos activos en los próximos 12 meses. Cada evento muestra el monto total a pagar y el activo. Click en el evento abre el proceso de distribución.
Totales por mes: suma de distribuciones a pagar en USD para planificación de tesorería del CFO.
Alerta 15 días antes de cada distribución: &apos;Se requiere fondeo del fideicomiso por $X USD para la distribución de [activo] el [fecha]&apos;. Enviada por email al CFO.
Exportable como CSV para consolidar en el modelo financiero del CFO.
7.1.2 Panel de Distribución del CFO (US-A-23, US-A-24)
Este es el proceso más crítico del sistema y el más expuesto a error humano. El panel guía al CFO a través de 4 pasos secuenciales con confirmación explícita en cada uno.
Paso 1 — Confirmación de pago propietario: botón con campo de comprobante obligatorio. Al confirmar, Formance calcula el waterfall automáticamente vía Numscript (US-L-15).
Paso 2 — Vista previa del waterfall: tabla con el monto a pagar a cada holder (calculado por Formance), el fee ACCET separado y la retención DIAN por holder colombiano. El CFO revisa antes de ejecutar.
Paso 3 — Firma del CFO: modal de confirmación doble que muestra el resumen del impacto (total a distribuir, número de holders, método de pago) + 2FA del CFO obligatorio.
Paso 4 — Ejecución batch: el Forwarder ejecuta el batch PSE/USDC con progreso visible en tiempo real (X/N holders pagados). Los TX IDs de Hedera aparecen en tiempo real por holder.
Gestión de fallos: si algún pago del batch falla, aparece en una bandeja de fallos con la causa específica (cuenta cerrada, wallet inválida, fondos insuficientes) y las opciones de resolución.
7.1.3 Proceso de Cierre al Vencimiento (US-A-26, US-A-27)
Checklist de cierre guiado: pago propietario confirmado ✓ · distribución final ejecutada ✓ · token burn ejecutado ✓ · asiento contable de cierre generado ✓ · activo archivado ✓. Cada paso requiere confirmación del responsable.
El botón &apos;Ejecutar burn&apos; está físicamente desactivado hasta que la distribución final tiene estado &apos;completada al 100%&apos;. Esta restricción de UI complementa la validación del Forwarder.
Post-burn: el activo pasa a estado &apos;Cerrado&apos; con el TX ID del burn visible. Los holders reciben notificación automática. El activo se archiva con historial completo preservado.
Asiento contable de cierre generado desde Formance: exportable en CSV compatible con SIIGO / World Office / Quickbooks.
7.1.4 Reportes Regulatorios (US-A-29, US-A-30)
Reporte SFC Sandbox: generado automáticamente con datos del sistema. El oficial de cumplimiento solo revisa y firma antes de enviar. Formato configurable según la plantilla SFC vigente actualizable sin intervención del CTO.
Medios magnéticos DIAN: consolidado por inversor en formato XML compatible con cargue directo en el portal DIAN. Disponible desde el 1 de enero del año siguiente al período fiscal.
Certificados de retención individuales: generados en lote (un PDF por inversor colombiano) descargables en ZIP. Firmados digitalmente por AgoraLink S.A.S. como agente retenedor.
Reporte ejecutivo directorio: PDF generado en menos de 2 minutos con KPIs vs proyección del modelo financiero MVP NdS 2026, semáforo de alerta y sección de acciones pendientes de decisión.
7.1.5 Gobernanza, Travel Rule y ESG (US-A-32, US-A-33, US-A-34)
Panel de gobernanza: configuración de roles, timelock (mínimo 24h para funciones críticas) y quórum (configurable: ej. 2 de 3 firmantes del directorio). Cada cambio de parámetro requiere aprobación del quórum + registro en audit trail.
Travel Rule GAFI: captura automática de originador y beneficiario para transferencias de tokens entre inversores que superen $1,000 USD. Integración con protocolo IVMS 101 para intercambio con VASPs contrapartes.
Reporte ESG: métricas ambientales (inmuebles con certificación verde, m²), sociales (propietarios PYME habilitados) y de gobernanza (% acciones on-chain auditables). Exportable como PDF one-pager en inglés y español para pitches internacionales.

1. Resumen Ejecutivo

1.1 Propósito del Documento
Este PRD define los requerimientos funcionales, lineamientos de implementación backend/frontend y el plan de tareas detallado para integrar Formance Ledger como el motor contable interno (capa privada) de ACCET. El documento cubre las 6 épicas técnicas de la Suite de User Stories 04/05 v2, estructuradas en 5 fases de implementación secuenciales.

Concepto arquitectónico clave:  ACCET opera un sistema Dual-Ledger. La capa pública (Hedera HTS) registra la verdad on-chain visible para todos. La capa privada (Formance Ledger) gestiona el libro contable interno: separación de fees, retenciones DIAN, waterfall financiero y cierre contable. Ambas capas deben permanecer ISOMÓRFICAS — lo que registra Hedera debe cuadrar exactamente con lo que registra Formance.

1.2 Problema que Resuelve
Sin Formance Ledger, ACCET enfrenta cinco riesgos críticos:
El CFO ejecuta el waterfall financiero manualmente en Excel — error humano garantizado en cada distribución semestral.
No existe separación automática del fee ACCET (3%) del capital de inversores — riesgo de mezcla de fondos.
La retención DIAN para inversores colombianos no se calcula automáticamente — pasivo fiscal no cuantificado.
No hay libro contable verificable para auditorías externas o due diligence de inversores institucionales.
Las distribuciones on-chain (Hedera) y los registros financieros internos son sistemas desconectados — reconciliación imposible.

4.1 Lineamientos Backend — Fase 1
4.1.1 Deploy Docker Compose (US-L-02)
El Formance Stack se despliega como un conjunto de servicios Docker. La configuración de compose debe separar explícitamente los entornos producción y staging.

Estructura de servicios Docker Compose requerida:  formance-ledger (API principal) · formance-worker (procesador de scripts) · postgresql:16 · redis (cola de scripts) · vault-agent (sidecar de secrets) · nextcloud-backup-cron (tarea de backup diario)

Variables de entorno: NUNCA en .env versionado. Inyectar desde HashiCorp Vault via vault-agent sidecar.
La instancia de staging debe usar datos ficticios sin conexión posible a producción (red Docker separada).
Cada push al repositorio activa el scanner git-secrets para detectar credenciales accidentales.
Backup diario de PostgreSQL Formance: pg_dump cifrado → Nextcloud NAS local + replica cloud secundario.
RPO objetivo: 24 horas. RTO objetivo: 4 horas. Prueba de restauración mensual obligatoria en staging.

4.1.2 Plan de Cuentas Numscript (US-L-03)
Las cuentas virtuales en Formance se crean mediante la primera transacción Numscript que las referencia. No existe un DDL para crearlas: se auto-crean al primer movimiento.

Script de inicialización requerido (Numscript):  send [USDC 0] (source = @world, destination = @accet:fees:operativos) — esto inicializa la cuenta con balance 0 sin mover fondos reales. Repetir para cada cuenta del plan.

Las cuentas @holder:{investor_id}:{activo_id} se crean automáticamente al aprobar el KYC del inversor para ese activo — no de forma global.
La cuenta @dian:retenciones:{año_fiscal} se crea al inicio de cada año fiscal con el Numscript de cierre anterior.

El tiempo máximo entre activar el botón y el efecto en ambos sistemas es 60 segundos.

Post-incidente: informe en 72h con causa raíz, qué capa fue afectada (Hedera/Formance/ambas) y medidas preventivas implementadas.
La revocación de clave Hedera sigue el procedimiento Shamir de recuperación: 2 de 3 fragmentos (CTO, CEO, Asesor Legal) en ubicaciones físicas separadas.

8.1.3 Suite de Regresión Dual-Ledger (US-L-36)
La suite de regresión valida los flujos críticos en AMBAS capas de forma coordinada. Un fallo en cualquier test de categoría MUST bloquea el despliegue en producción:

Test de integración completo: pago PSE confirmado → Numscript ejecutado en Formance → token emitido en Hedera testnet.
Test de distribución: Numscript distribución ejecutado → webhook al Forwarder → distribución USDC en Hedera testnet → balances actualizados en Formance.
Test de reconciliación: balances Formance deben cuadrar con balances Mirror Node al finalizar cada test. Cero discrepancias aceptadas.
Test de rollback: simular fallo en el Paso 2 del despliegue coordinado — verificar que el token HTS se pausa automáticamente.
Test de atomicidad: introducir un fallo forzado en el Numscript — verificar que todos los balances regresan al estado previo.

1. Resumen Ejecutivo
1.1 Propósito del Documento
Este PRD define los requerimientos funcionales, el diseño técnico y el plan de trabajo del Forwarder Ledger: el servicio backend propio de ACCET que actúa como puente isomórfico entre Formance Ledger (motor contable interno) y Hedera Hashgraph (registro público on-chain). Es el cuarto PRD de la serie Dual-Ledger.
El Forwarder Ledger es el activo tecnológico más estratégico de ACCET. No es un adaptador genérico ni una integración de terceros: es el código propietario que implementa el isomorfismo contable del sistema — la garantía de que la verdad financiera interna (Formance) y la verdad pública verificable (Hedera) son siempre idénticas.
A diferencia de Formance y Hedera — que son tecnologías de terceros que ACCET configura — el Forwarder Ledger es código que ACCET escribe, versiona y opera. Eso lo convierte en el moat tecnológico central del negocio.
1.2 Problema que Resuelve
Sin el Forwarder, la ejecución on-chain de cada operación (emisión, distribución, burn) requiere intervención manual del CTO — no es escalable más allá de 3 activos simultáneos.
Sin idempotencia garantizada, un reintento de pago PSE o un webhook duplicado de Formance resultaría en doble emisión de tokens — un incidente financiero grave y de reputación.
Sin cola durable, una caída temporal de Hedera mainnet durante una distribución semestral haría perder los eventos en vuelo y el CFO tendría que reconstruir manualmente qué holders no fueron pagados.
Sin reconciliación automática, la divergencia entre Formance y Hedera solo se detectaría en una auditoría externa — cuando el daño ya está hecho.
Sin el score de isomorfismo, el equipo directivo no tiene un indicador único de la salud del sistema dual-ledger para reportar a inversores institucionales.

2. Arquitectura del Forwarder Ledger
2.1 Posición en el Sistema Dual-Ledger
El Forwarder Ledger ocupa la capa intermedia del sistema. Nunca es llamado directamente por el usuario final ni por el propietario o el inversor. Solo recibe instrucciones del Admin Dashboard y webhooks de Formance, y solo ejecuta acciones en Hedera.

⬡  Formance Ledger
Fuente de eventos
Ejecuta Numscripts. Al confirmar un movimiento financiero, envía un webhook al Forwarder con el evento y el formance_tx_id.

Puente isomórfico (código ACCET)
Recibe webhook · valida schema · persiste en cola BullMQ · firma TX Hedera · confirma en Formance · reconcilia.
◈  Hedera HTS
Destino de las TXs
Recibe TXs firmadas del Forwarder. Registra el hecho on-chain de forma inmutable.

4. Fase 1 — Fundamentos y Conexiones Dual
4.1 Lineamientos Backend — Fase 1
4.1.1 Conexiones a Formance y Hedera (US-F-01)
El Forwarder arranca con un health check obligatorio que verifica las tres conexiones críticas. Si cualquiera falla, el servicio no arranca y escala una alerta P1 inmediatamente.
Configuración Formance: FORMANCE_BASE_URL + JWT token con scope de escritura. Test de ping: GET /api/ledger/_info debe responder en menos de 200ms.
Configuración Hedera: HEDERA_ACCOUNT_ID + HEDERA_PRIVATE_KEY leída del Vault en runtime (no en variables de entorno). Test: AccountBalanceQuery en menos de 2 segundos.
Configuración Redis: REDIS_URL para BullMQ. Test: PING debe responder PONG. Si Redis no está disponible al arrancar, el Forwarder falla con error explícito.
El indicador de entorno activo — MAINNET o TESTNET — se estampa en TODOS los logs y en CADA mensaje de la cola para hacer imposible la confusión entre ambientes.
4.1.2 Receptor de Webhooks de Formance con HMAC (US-F-02, US-F-08)
El endpoint de webhooks es la única puerta de entrada del Forwarder. Todo evento financiero pasa por aquí antes de entrar a la cola.
Cada webhook de Formance incluye el header X-Formance-Signature con una firma HMAC-SHA256 del cuerpo usando el WEBHOOK_SECRET compartido. El Forwarder verifica esta firma antes de cualquier procesamiento.
Un webhook con firma inválida recibe HTTP 401 y se registra como intento no autorizado en el audit trail. No se encola nada.
Validación del schema JSON contra el contrato de mensajes versionado (Sección 2.3). Un mensaje con campos faltantes recibe HTTP 400 con detalle del campo inválido.
Al pasar ambas validaciones: el evento se persiste en la cola BullMQ en menos de 100ms y el Forwarder responde HTTP 202 Accepted a Formance. Formance interpreta el 202 como &apos;en procesamiento&apos;.
4.1.3 Cola BullMQ con Persistencia Durable (US-F-03)
Principio de diseño: Un evento NUNCA se procesa sin estar primero en la cola. Persist-first, execute-second. Si el Forwarder se cae entre recibir el webhook y encolar, Formance puede reenviar el webhook y la idempotencia por formance_tx_id lo manejará.
Redis configurado con AOF (Append-Only File) habilitado para durabilidad. Un reinicio del servidor Redis no pierde los jobs encolados.
Los jobs en estado &apos;active&apos; (ejecutándose) al momento de un crash del Forwarder son movidos automáticamente a &apos;stalled&apos; por BullMQ y re-intentados al reiniciar.
Estructura de cada job: event_type · asset_id · payload completo del webhook · formance_tx_id · prioridad · intentos · próximo_reintento · environment (MAINNET/TESTNET).
4.1.4 Idempotencia por formance_tx_id (US-F-12)
Al encolar un evento: el Forwarder verifica en Redis si el formance_tx_id ya fue procesado (TTL de 30 días). Si existe: responde HTTP 200 con el hedera_tx_id original. No encola ni ejecuta nada.
Al completar un job exitosamente: el Forwarder escribe en Redis: SET idempotency:{formance_tx_id} {hedera_tx_id} EX 2592000 (30 días).
Esta operación protege contra: reenvíos de webhook por timeout de red, reintentos automáticos de Formance, y fallos de confirmación donde Formance no recibió el HTTP 202.
4.1.5 Priorización y Entornos (US-F-05, US-F-04)
BullMQ permite asignar prioridad numérica a cada job. El Forwarder usa: P0=1 (emergency_pause), P1=10 (token_burn en vencimiento), P2=100 (token_emit, usdc_distribute), P3=1000 (whitelist_add/remove, upgrades).
El selector de entorno es una variable de arranque, no configurable en caliente. Para cambiar de testnet a mainnet se requiere un redeploy completo con la variable de entorno HEDERA_NETWORK=mainnet. Esto hace imposible el cambio accidental de entorno.
Los logs de mainnet y testnet van a índices Loki separados: accet-forwarder-mainnet y accet-forwarder-testnet. Los dashboards de Grafana tienen un filtro de entorno obligatorio en la barra superior.

5. Fase 2 — Resiliencia y Modos Degradados
5.1 Lineamientos Backend — Fase 2
5.1.1 Modo Degradado ante Caída de Hedera (US-F-24)
Hedera mainnet es un servicio externo. Las caídas, aunque raras, ocurren. El Forwarder debe manejarlas sin pérdida de eventos y sin intervención humana de emergencia.
Detección automática: 3 TX fallidas consecutivas por timeout de red activa el modo degradado de Hedera. El Forwarder sigue aceptando webhooks de Formance y encolando jobs, pero pausa la ejecución hacia Hedera.
Polling de recuperación: cada 60 segundos el Forwarder ejecuta un AccountBalanceQuery de prueba. Al recibir una respuesta exitosa, reanuda la ejecución de la cola en orden de prioridad.
El Admin Dashboard muestra un banner prominente: &apos;Forwarder en modo degradado — Hedera no disponible desde [hora]&apos;. El banner desaparece al recuperarse la red.
SLA de jobs en modo degradado: los jobs pueden esperar hasta 48 horas en la cola antes de marcarse como fallidos definitivos y escalar al CTO.
5.1.2 Modo Degradado ante Caída de Formance (US-F-25)
Health check a la API de Formance cada 30 segundos. Si falla 2 veces consecutivas: alerta P1 urgente al Ledger Operator y al CTO.
Jobs que requieren confirmar el TX ID en Formance (todos excepto whitelist_add/remove) se pausan hasta que el servicio se recupere.
Los jobs de whitelist_add/remove pueden continuar si solo Formance está caído, ya que no requieren confirmación en Formance para ejecutarse en Hedera.
5.1.3 Política de Reintentos con Backoff Exponencial (US-F-23)
Errores retriables: timeout de red, HBAR insuficiente temporal, nodo Hedera no disponible. Política: intento 1 inmediato → 2: 30s → 3: 2min → 4: 10min → 5: 1h → 6+: cada 6h. Máximo 10 intentos.
Errores no retriables: AccountID Hedera inválido, token no asociado en la cuenta destino, rechazo por compliance ERC-3643 (account not in whitelist). Estos se marcan como fallos definitivos inmediatamente.
Al superar 3 intentos fallidos: alerta P2 al Ledger Operator con el detalle del error. Al llegar al intento 10 sin éxito: alerta P1 al CTO.
Throttle de concurrencia: máximo 5 TXs simultáneas hacia Hedera. Configurable desde el Admin Dashboard sin reiniciar el Forwarder.
5.1.4 Plan de Disaster Recovery (US-F-41)
Redis con persistencia AOF + snapshot RDB cada 15 minutos hacia Nextcloud NAS. La cola de jobs sobrevive la pérdida completa del servidor del Forwarder.
Snapshots del estado del Forwarder (imagen Docker + variables de configuración sin secrets) cada 6 horas hacia proveedor cloud secundario.
Procedimiento de restauración documentado y verificado: RTO (Recovery Time Objective) de 2 horas. El Forwarder restaurado retoma la cola de Redis exactamente desde donde la dejó.
Prueba de DR semestral obligatoria en staging con documentación del tiempo real de recuperación.

6. Fase 3 — Operaciones de Tokens Completas
6.1 Lineamientos Backend — Fase 3
6.1.1 Emisión Automática de Tokens (US-F-15 + US-F-08)
La emisión es el primer flujo end-to-end en producción. El tiempo desde la confirmación del pago (Formance) hasta que el inversor ve su token (Hedera) es el KPI más visible del sistema para los inversores.
Flujo completo: PSE/USDC confirmado → Formance ejecuta Numscript de ingreso → webhook token_emit → Forwarder valida HMAC + schema + idempotencia → job encolado P2 → clave Operator obtenida del Vault en runtime → TokenTransferTransaction(treasury → investor_account) firmada y enviada → TX confirmada en Mirror Node → TX ID registrado en Formance → inversor ve tokens en el dashboard.
Tiempo objetivo: menos de 30 segundos desde el webhook hasta la confirmación en Hedera. Medido en Grafana por el percentil P95.
Pre-check de whitelist: antes de construir la TX, el Forwarder verifica que el investor_hedera_id está en la whitelist del Identity Registry del contrato ERC-3643 del activo. Si no está: error no retriable + alerta al equipo de Compliance.
6.1.2 Distribución Batch USDC (US-F-16 + US-F-09)
La distribución batch es el proceso financiero más crítico. El Forwarder recibe la lista exacta de Formance y ejecuta sin recalcular.
Lotes de hasta 10 transferencias USDC por TransferTransaction de Hedera para optimizar el consumo de HBAR. Una distribución a 30 holders genera exactamente 3 TXs.
Progreso del batch disponible en el endpoint GET /distribution/{distribution_id}/progress en tiempo real: holders_total, holders_pagados, holders_pendientes, holders_fallidos, txids[].
Si una TX del batch falla: solo los holders de esa TX van a la cola de reintento. Los holders de TXs exitosas no se repiten jamás (idempotencia por holder dentro del batch via Redis).
Post-distribución: Formance recibe la confirmación con la lista completa de TX IDs por holder. El asiento contable en Formance queda cerrado solo cuando todos los holders tienen su TX ID confirmado.
6.1.3 Token Burn (US-F-17 + US-F-10)
Pre-check obligatorio antes del burn: el Forwarder llama a la API de Formance para verificar que sum(@holder:*:{asset_id}) == 0. Si hay cualquier balance pendiente: error 409 al Admin Dashboard con detalle.
TokenBurnTransaction firmada con la Treasury Account (clave obtenida del Vault al momento de la firma). Supply objetivo: 0.
Post-burn: consulta al Mirror Node para confirmar supply = 0. Solo si el Mirror confirma supply = 0, el Forwarder reporta éxito al Admin Dashboard y cierra el ciclo en Formance.
El TX ID del burn se registra en: el HCS topic del activo, el expediente en Formance, el Admin Dashboard y la notificación automática a todos los holders.
6.1.4 Gestión de Whitelist ERC-3643 (US-F-18 + US-F-11)
whitelist_add: ContractExecuteTransaction que llama a addIdentity(hedera_account_id, country_code) del Identity Registry. La firma es del Operator Account.
whitelist_remove: llamada a removeIdentity(). Requiere validación previa: si el inversor tiene tokens activos en el activo, el Forwarder rechaza el remove con error 409.
TX ID de cada operación de whitelist registrado en el expediente del inversor en Formance. Visible en el panel de whitelist del Admin Dashboard.
6.1.5 Pausa de Emergencia (US-F-13, US-F-14)
El evento emergency_pause recibe prioridad P0 (valor 1 en BullMQ). Al recibirlo: el Forwarder mueve TODOS los jobs pendientes a estado &apos;suspended&apos; (no eliminados), y ejecuta ContractExecuteTransaction → pause() en cada contrato activo de forma secuencial.
Tiempo objetivo de pausa completa: menos de 60 segundos para una cartera de hasta 10 activos activos simultáneos.
Reanudación post-incidente: el evento de reanudación requiere doble firma CTO + CEO desde el Admin Dashboard. Al recibir la reanudación: el Forwarder ejecuta una reconciliación completa antes de desencolar. Si la reconciliación detecta diferencias: no se reanuda y se alerta con detalle.

7. Fase 4 — Reconciliación Isomórfica
7.1 Lineamientos Backend — Fase 4
7.1.1 Ciclo de Reconciliación Automático Diario (US-F-29)
La reconciliación isomórfica es la función más importante del Forwarder a largo plazo. En el corto plazo garantiza la integridad del sistema. En el mediano plazo es el SLA que ACCET puede publicar a inversores institucionales.
Ciclo automático diario a las 2:00 AM UTC (fuera del horario de mayor actividad). Para cada activo activo: consulta Mirror Node y compara contra Formance en tres dimensiones.
Dimensión 1 — Supply: supply_on_chain (Mirror Node TokenInfo) debe ser igual a sum(@holder:*:{asset_id}) en Formance. Tolerancia: 0.
Dimensión 2 — Distribuciones: cada TX ID de distribución registrado en Hedera debe tener su Numscript correspondiente en Formance con la referencia cruzada. Y viceversa: cada Numscript de distribución en Formance debe tener su TX ID en Hedera.
Dimensión 3 — Balance USDC: suma de USDC en las cuentas Hedera de los holders debe igualar la suma de @holder:{id}:{asset_id} en Formance para ese activo.

7.1.3 Score de Isomorfismo (US-F-32)
Score = (activos_reconciliados_sin_diferencia / total_activos_reconciliados) × 100. Publicado en el endpoint GET /reconciliation/score como un número entre 0 y 100 con dos decimales.
Target operacional: 100.00. Cualquier diferencia baja el score por debajo de 100.
SLA publicable a inversores institucionales: &apos;ACCET garantiza isomorfismo del 99.9% medido en ciclos diarios de reconciliación.&apos;
El score está visible en la barra superior del Admin Dashboard en todo momento (consumido vía WebSocket para actualización en tiempo real post-ciclo de reconciliación).
7.1.4 Reconciliación Bajo Demanda (US-F-34)
Endpoint POST /reconciliation/{asset_id}/run disponible para el Admin Dashboard. El CFO puede disparar una reconciliación bajo demanda antes de ejecutar una distribución.
Tiempo objetivo de reconciliación bajo demanda para un activo con hasta 50 holders: menos de 30 segundos.
Si la reconciliación bajo demanda detecta diferencias: el endpoint GET /distribution/{id}/execute queda bloqueado para ese activo hasta que se resuelva la diferencia.

8. Fase 5 — Testing, Seguridad y Documentación
8.1 Lineamientos Backend — Fase 5
8.1.1 Suite de Regresión Dual-Ledger (US-F-36, US-F-37)
La suite de regresión valida los flujos críticos en ambas capas de forma coordinada. Un fallo en cualquier test bloquea el despliegue en mainnet hasta que se resuelva.
Test E2E de emisión: PSE confirmado en Formance staging → webhook token_emit → TokenTransferTX en Hedera testnet → TX ID registrado en Formance → balance Mirror Node coincide con Formance. Tiempo máximo: 60 segundos.
Test de distribución batch: Numscript de distribución en Formance → webhook usdc_distribute → lotes de USDC en testnet → TX IDs por holder → reconciliación verifica isomorfismo.
Test de idempotencia: enviar el mismo webhook dos veces. El segundo debe recibir HTTP 200 con el TX ID original. Solo una TX debe existir en Hedera testnet.
Test de rollback de distribución: simular fallo en la TX del segundo lote. Verificar que los holders del primer lote no se repagan y los del segundo lote van a reintento.
Test de modo degradado: simular Hedera testnet no disponible. Verificar que los webhooks se aceptan y la cola crece. Restaurar Hedera. Verificar que los jobs se procesan en orden.
Test de reconciliación: introducir una discrepancia Tipo B manualmente. Verificar que el ciclo la detecta, la clasifica correctamente y genera la alerta esperada.
Tiempo máximo de la suite completa: 15 minutos en CI/CD.
8.1.2 Secrets Management y Seguridad (US-F-41)
Regla absoluta: Las claves privadas Hedera y los JWT de Formance NUNCA se almacenan en el servidor del Forwarder. Se leen del Vault en cada operación que lo requiere y se descartan de la memoria después del uso.
Scanner git-secrets + truffleHog en cada push al repositorio Gitea. Los patrones de detección cubren: claves privadas Hedera (DER base64), tokens JWT de Formance, strings de conexión Redis con contraseña.
Redacción automática en Pino: cualquier campo en el log que contenga &apos;key&apos;, &apos;secret&apos;, &apos;token&apos;, &apos;password&apos;, &apos;private&apos; se reemplaza por [REDACTED] antes de escribir al log.
Rotación de secrets: cuando el Vault rota una clave o token, el Forwarder la lee en la siguiente operación automáticamente sin reinicio. Esto se logra leyendo del Vault en cada firma, no al arrancar.
8.1.3 Documentación Técnica (US-F-07, US-F-38, US-F-39, US-F-40)
OpenAPI 3.0 completo del Forwarder: todos los endpoints con parámetros, respuestas exitosas, respuestas de error, schemas de mensajes y ejemplos de curl. Publicado internamente en la wiki de Gitea.
Diagrama de secuencia por tipo de evento: Formance webhook → Forwarder queue → Hedera TX → confirmación Formance. Un diagrama por cada uno de los 7 tipos de evento.
Runbook de incidentes (mínimo 4): distribución batch a mitad con jobs fallidos · cola creciendo sin procesar · reconciliación con diferencia Tipo C · Forwarder no responde. Cada runbook: síntoma → diagnóstico → resolución → escalamiento.

1. Resumen Ejecutivo
1.1 Propósito del Documento
Este PRD define los requerimientos funcionales, lineamientos de implementación backend/frontend y el plan de trabajo para integrar Hedera Hashgraph como la capa pública on-chain del sistema ACCET. El documento es el segundo en la serie de PRDs de la arquitectura Dual-Ledger y se apoya directamente sobre el PRD de Formance Ledger (v1.0) ya aprobado.
Hedera actúa como el notario digital del sistema: registra cada hecho financiero de forma inmutable, verificable de forma independiente por cualquier inversor o regulador sin depender de ACCET como intermediario de confianza.
1.2 Problema que Resuelve
Sin la capa Hedera, el sistema ACCET enfrenta cinco riesgos críticos no resueltos por Formance:
Los inversores no tienen prueba independiente de que sus tokens existen y sus distribuciones fueron ejecutadas — solo pueden confiar en los reportes de ACCET.
Los documentos legales del activo (escritura, avalúo, contrato fideicomiso) no tienen timestamp inmutable verificable — pueden ser alterados sin evidencia.
El compliance de transferencias entre inversores no está automatizado — sin ERC-3643, un token podría transferirse a un wallet no verificado.
El precio del inmueble (LTV) no está alimentado por un oracle independiente — el equipo de riesgo depende de actualizaciones manuales.
La prueba de auditoría D2 del audit de la Matriz RWA (score 38/70 — Crítico) no puede cerrarse sin contratos verificados on-chain.

4. Fase 1 — Infraestructura y Conexión Hedera
4.1 Lineamientos Backend — Fase 1
4.1.1 Configuración de la Conexión Mainnet (US-L-01)
La conexión con Hedera mainnet se establece usando el SDK oficial @hashgraph/sdk en su versión LTS. La instancia de testnet se mantiene siempre activa como entorno de staging para validar cambios antes de producción.
La variable HEDERA_NETWORK determina el entorno activo. Los valores permitidos son exactamente: mainnet, testnet, previewnet. Cualquier otro valor rechaza el inicio del servicio.
HEDERA_ACCOUNT_ID y HEDERA_PRIVATE_KEY se inyectan desde HashiCorp Vault via vault-agent sidecar al arrancar el Forwarder. Nunca en archivos .env versionados.
Test de conectividad obligatorio al arrancar: AccountBalanceQuery ejecutado y confirmado en menos de 2 segundos. Si falla, el servicio no arranca y escala alerta P1 al CTO.
Buffer de HBAR operacional: la treasury debe tener mínimo 100 HBAR antes de habilitar la emisión en producción. El Forwarder verifica este balance en el arranque.
4.1.2 Cuentas Hedera Operadoras (US-L-03 / US-L-05)
Las tres cuentas operadoras se crean de forma independiente con claves distintas y se registran en Vault antes del primer despliegue. La creación se ejecuta una sola vez en mainnet; en testnet se usan cuentas separadas con fondos de prueba.
Treasury Account: creada con AccountCreateTransaction. La clave se almacena en Vault bajo el scope treasury-key con acceso exclusivo del CTO (política Vault: solo puede read el rol CTO).
Fee Collector Account: creada de la misma forma. Clave en Vault scope fee-key, política de acceso: roles CTO y CFO.
Operator Account: clave en Vault scope operator-key, acceso exclusivo del Ledger Operator (rol del servicio Forwarder Ledger).
Registro en Gitea: AccountIDs de las tres cuentas (solo los IDs, jamás las claves) se documentan en el repositorio de infraestructura para trazabilidad.
Importante: La rotación de claves se planifica cada 180 días mediante AccountUpdateTransaction. El procedimiento completo está documentado en el runbook del Forwarder (US-L-39, US-F-40).
4.1.3 Mirror Node y Entornos (US-L-04 / US-L-07)
Mirror Node API para consultas de estado: mainnet.mirrornode.hedera.com para producción, testnet.mirrornode.hedera.com para staging. Las consultas de lectura son gratuitas — se prefieren sobre las ConsensusQuerys de pago.
Cache de 30 segundos en Redis para consultas frecuentes de balance. Invalidación automática al detectar nueva TX del activo en el Mirror Node.
El selector de entorno es una función crítica de seguridad. Un badge visual con fondo verde (PRODUCCIÓN) o amarillo (STAGING) debe aparecer en el Admin Dashboard en todo momento cuando el Ledger Operator está operando.
Los logs de mainnet y testnet van a streams separados en Grafana Loki. Nunca en el mismo índice.
4.2 Lineamientos Frontend — Fase 1 (Insumos para Admin Dashboard PRD)
Handoff Admin Dashboard: Los siguientes widgets son los primeros que el PRD del Admin Dashboard deberá especificar. Se documentan aquí para garantizar que la capa Hedera expone los datos correctos.
Widget de estado de Hedera: red (mainnet/testnet), latencia de la última TX, balance HBAR de treasury y operator accounts. Polling vía WebSocket cada 60 segundos. Alerta visual si balance treasury &lt; 50 HBAR.
Badge de entorno activo: VERDE = PRODUCCIÓN, AMARILLO = STAGING. Visible en la barra superior del Admin Dashboard en todo momento cuando el operador tiene sesión activa.
Panel de Mirror Node: estado de la conexión (online/degradado/offline), latencia de consulta promedio, último timestamp de bloque procesado.

5. Fase 2 — Smart Contracts ERC-3643 y Despliegue
5.1 Lineamientos Backend — Fase 2
5.1.1 Parametrización y Generación del Contrato (US-L-08)
Cada activo tokenizado requiere su propio contrato ERC-3643 T-REX adaptado a Hedera HTS. Los parámetros del contrato se extraen automáticamente del expediente aprobado por el comité para eliminar errores de transcripción manual.
Plantilla ERC-3643 con variables: nombre del token, símbolo, supply total, compliance module address, identity registry address. El sistema pre-llena estos campos desde el expediente del activo al crear el contrato.
El contrato ERC-3643 se compone de tres módulos obligatorios en ACCET: Identity Registry (whitelist de inversores verificados por KYC), Compliance Module (restricciones por jurisdicción), Transfer Manager (reglas de pausa y restricción de transferencias).
Hash SHA-256 del bytecode generado y registrado en el expediente del activo ANTES del despliegue. Permite verificar en cualquier momento que el contrato desplegado es el que fue aprobado.
El botón de despliegue en el Admin Dashboard queda físicamente bloqueado hasta que el tracker de auditoría externa tiene estado Aprobado. Esta es una restricción de UI, no solo de negocio.
5.1.2 Proceso de Auditoría Externa (US-L-10)
La auditoría de smart contracts es el cierre de la brecha más crítica del dominio D2 del audit de la Matriz RWA (score 38/70 — el peor del sistema). Sin este paso, el despliegue en mainnet está bloqueado.
Flujo de solicitud de auditoría: el CTO genera un repo privado en Gitea con el código fuente del contrato + contexto del activo. Se comparte acceso de solo lectura al auditor externo certificado.
Tracker de auditoría con estados: Solicitada → En revisión → Hallazgos recibidos → Corregido → Aprobado. Cada transición de estado requiere documentación adjunta.
Hallazgos clasificados por severidad: Crítico (bloquea despliegue), Alto (requiere corrección antes de desplegar), Medio (corrección en siguiente versión), Bajo (documenta y acepta).
El reporte de auditoría aprobado se almacena en el expediente del activo en Nextcloud NAS y se adjunta como metadata del contrato en Formance.
Costo estimado: Auditoría de smart contract ERC-3643: USD $3,000–$8,000 dependiendo del auditor. Incluir en el presupuesto de la Fase 2 post-seed.
5.1.3 Despliegue Coordinado Hedera + Formance (US-L-11)
El despliegue de un nuevo activo sigue una secuencia estricta de tres pasos. Si cualquier paso falla, el rollback es automático. La coordinación con el PRD de Formance garantiza que ambas capas se activan juntas.

5.1.4 Upgrades con Timelock (US-L-12)
Toda actualización de un contrato en producción requiere una propuesta on-chain con hash del nuevo bytecode y descripción del cambio. El timelock mínimo es de 48 horas desde la propuesta hasta la ejecución.
Durante el timelock, todos los holders del activo reciben una notificación automática con el detalle del cambio propuesto y la fecha de aplicación.
La ejecución del upgrade (contrato Hedera + Numscript actualizado en Formance) requiere doble firma: CTO + CEO. Sin ambas firmas, la ejecución está bloqueada aunque el timelock haya vencido.
El upgrade puede cancelarse durante el período de timelock por quórum del directorio (2 de 3: CEO, CFO, CTO). La cancelación se registra on-chain.

Diff visual de Numscripts: vista de cambios entre versiones (líneas añadidas en verde, eliminadas en rojo) antes de aprobar un upgrade. Reutilizar el componente de diff del expediente de contratos.

6. Fase 3 — Operaciones de Tokens
6.1 Lineamientos Backend — Fase 3
6.1.1 Emisión de Tokens Post-Pago (US-L-15 + Suite 05 US-F-08, US-F-15)
La emisión de tokens es el primer flujo end-to-end del sistema dual-ledger en producción. El Forwarder Ledger actúa como intermediario: recibe el webhook de Formance (pago confirmado) y construye la TokenTransferTransaction de Hedera.
Flujo completo: PSE/USDC confirmado → Formance ejecuta Numscript de ingreso → webhook al Forwarder con event_type: token_emit → job encolado en BullMQ → TransferTransaction treasury → investor_account firmada y enviada → TX ID registrado en Formance y en el expediente del inversor.
Tiempo objetivo de emisión: menos de 30 segundos desde la confirmación del pago hasta que el inversor ve sus tokens en el dashboard.
ERC-3643 compliance: antes de ejecutar cualquier TransferTransaction, el Forwarder verifica que el AccountID del inversor está en la whitelist del Identity Registry. Si no está, la TX es rechazada y se escala al equipo de cumplimiento.
Idempotencia garantizada por el formance_tx_id como clave única en Redis (TTL 30 días). Un reintento de webhook de Formance por fallo de red nunca resulta en doble emisión.
6.1.2 Distribución Batch USDC (US-L-16 + US-F-09, US-F-16)
La distribución semestral es el proceso financiero más crítico del sistema. Formance calcula el waterfall exacto (fee, holders proporcional, retención DIAN) y el Forwarder ejecuta el batch on-chain sin recalcular — usa los montos exactos de Formance.
Batch de hasta 10 transferencias USDC por TransferTransaction de Hedera para optimizar consumo de HBAR. Una distribución a 30 holders genera 3 TXs batch.
TX ID único registrado por holder en la base de datos del Forwarder y en el registro de Formance de ese holder. El inversor puede verificar su TX ID en hashscan.io de forma independiente.
Si una TX del batch falla: los holders de esa TX específica van a cola de reintento con backoff exponencial. Los holders de TXs exitosas no se repiten (idempotencia por holder).
Progreso del batch visible en tiempo real en el Admin Dashboard: X/N holders pagados — campo requerido por el Panel de Distribución del CFO (insumo Admin Dashboard PRD).
6.1.3 Token Burn al Vencimiento (US-L-17 + US-F-10, US-F-17)
El burn es el cierre definitivo del activo on-chain. Solo se ejecuta cuando Formance certifica que el balance de todos los holders es cero — todos recibieron su capital y rendimientos.
Verificación pre-burn obligatoria: el Forwarder consulta a Formance que sum(@holder:*:{activo_id}) == 0. Si hay algún balance pendiente, el burn se rechaza con error 409 y se alerta al CFO.
TokenBurnTransaction firmada con la treasury account. Supply resultante verificado contra el Mirror Node (debe ser exactamente 0) antes de reportar éxito al Admin Dashboard.
TX ID del burn registrado en: el expediente del activo, el HCS topic del activo, en Formance como el evento de cierre, y enviado como notificación a todos los holders.
6.1.4 Whitelist ERC-3643 (US-L-18 + US-F-11, US-F-18)
El Identity Registry se actualiza mediante ContractExecuteTransaction que llama a addIdentity(hedera_account, country_code). Esta TX se dispara automáticamente al aprobar el KYC del inversor en el Admin Dashboard.
El country_code es crítico para el Compliance Module: determina las reglas de restricción (US persons no acreditados, screening OFAC Venezuela).
removeIdentity() se ejecuta con causa registrada obligatoriamente en Formance. Requiere validación de que el inversor no tiene tokens pendientes de distribución antes de ejecutar la remoción.
6.2 Lineamientos Frontend — Fase 3 (Insumos para Admin Dashboard PRD)
Panel de emisión en tiempo real: tabla con últimas suscripciones confirmadas, TX ID Hedera de cada emisión con link a hashscan.io, tiempo transcurrido desde pago hasta token recibido. Meta: &lt; 30 segundos visible.
Panel de distribución batch del CFO: progreso en tiempo real X/N holders pagados, TX IDs por holder, opción de replay para holders fallidos (requiere 2FA del Ledger Operator).
Badge on-chain por posición del inversor en el dashboard del Holder: &apos;◈ Verificado en Hedera · [TX ID]&apos;. El Admin Dashboard PRD deberá especificar cómo este badge es consultado y renderizado.
Panel de whitelist: lista de inversores habilitados con AccountID Hedera, fecha de alta y TX ID de la operación de whitelist. Accesible solo para roles Compliance y CTO.

7. Fase 4 — Oracle Chainlink y HCS
7.1 Lineamientos Backend — Fase 4
7.1.1 Chainlink Price Feeds — LTV y TRM COP/USD (US-L-22, US-L-23)
Los Price Feeds de Chainlink son la fuente de verdad compartida entre la capa Hedera (on-chain) y la capa Formance (libro contable interno). Un único feed address debe alimentar a ambas capas para garantizar consistencia.
El feed address del LTV del inmueble se configura en el contrato ERC-3643 del activo (lectura on-chain por el smart contract y verificable por inversores en hashscan.io) Y como parámetro de Formance para los cálculos contables.
Fallback si el feed LTV no actualiza en 48 horas: usar el último precio conocido marcado como stale + alerta de datos obsoletos en el panel de riesgo del Admin Dashboard.
El feed COP/USD tiene heartbeat de 24 horas. Formance almacena el histórico de TRM para reconstruir certificados fiscales de años anteriores (requerido por DIAN).
Si el feed COP/USD no está disponible en Chainlink: fallback a API del Banco de la República con aviso explícito de fuente alternativa en todos los reportes generados durante ese período.
7.1.2 Hedera Consensus Service — HCS (US-L-25, US-L-26)
HCS proporciona el timestamp inmutable y descentralizado para los eventos del ciclo de vida del activo. Cada activo tiene su propio HCS topic creado al aprobarse el activo.
Mensajes HCS estructurados en JSON: {event_type, timestamp, actor, formance_tx_id, hedera_tx_id, hash (si aplica)}. El schema JSON está versionado en Gitea.
Eventos registrados en el HCS topic del activo: activo_aprobado, token_emitido, distribucion_ejecutada (con referencias cruzadas a Formance y Hedera), documento_actualizado, activo_cerrado.
Sellado de documentos: el hash SHA-256 del PDF del documento legal se envía al HCS topic con el sequence number como prueba de timestamp. El link a hashscan.io del mensaje HCS se incluye en el Legal Pack descargable por inversores.
El visor del timeline HCS debe aparecer en la ficha del activo en el marketplace del Holder (insumo para el Admin Dashboard PRD): &apos;Ver historial on-chain completo&apos;.
7.2 Lineamientos Frontend — Fase 4 (Insumos para Admin Dashboard PRD)
Panel de feeds Chainlink en Admin: nombre del feed, valor actual, timestamp de última actualización, desviación % vs anterior, estado de heartbeat (verde/amarillo/rojo). TRM del día visible con timestamp del oracle.
Herramienta de sellado de documentos: upload del PDF → hash SHA-256 calculado en el browser → envío al HCS topic del activo seleccionado → confirmación del sequence number y link a hashscan.io. Solo accesible para roles Ops y Legal.
Timeline on-chain del activo: visor de eventos HCS para el Holder en la ficha del activo con cada evento del ciclo de vida y su timestamp verificable. El Admin Dashboard PRD deberá definir el componente de visor HCS.
Reconciliación mensual: pantalla con resultado &apos;Todo concilia ✓&apos; o lista de discrepancias con detalle, tipo de diferencia (A/B/C) y botón de resolución guiada. Accesible solo para Ledger Operator y CTO.

8. Fase 5 — Seguridad, Disaster Recovery y Testing
8.1 Lineamientos Backend — Fase 5
8.1.1 HSM y Rotación de Claves (US-L-29, US-L-30)
REGLA ABSOLUTA: Ninguna clave privada de Hedera puede existir en texto plano en código, logs, variables de entorno o base de datos. Incumplir esta regla es un incidente de seguridad P1 inmediato.
HashiCorp Vault gestiona todos los secrets con scopes separados: scope treasury-key (solo CTO), scope operator-key (Ledger Operator y Forwarder), scope fee-key (CFO + CTO).
Rotación de claves cada 180 días: AccountUpdateTransaction con la nueva clave antes de revocar la anterior. El Forwarder entra en modo mantenimiento durante la ventana de rotación (máximo 15 minutos).
Scanner automático git-secrets + truffleHog en cada push al repositorio Gitea. Los patrones de detección incluyen claves privadas Hedera (formato DER), JWT tokens de Formance y strings Base64 de longitud característica de claves ECDSA.
Redacción automática en logs: cualquier string que coincida con el patrón de clave privada Hedera o JWT de Formance se reemplaza por [REDACTED] antes de escribir al log. Nunca se confía en que el desarrollador recuerde hacerlo manualmente.
8.1.2 Shamir Secret Sharing y Plan de Compromiso (US-L-31, US-L-32)
Backup de claves con Shamir Secret Sharing: 3 fragmentos, necesarios 2 de 3 para reconstruir. Fragmentos custodiados por CTO, CEO y Asesor Legal en ubicaciones físicas separadas.
Prueba de recuperación trimestral en testnet: reconstituir la clave desde 2 de 3 fragmentos, ejecutar una TX de prueba y verificar que el proceso toma menos de 60 minutos. Resultado documentado en audit trail.
Plan de respuesta a compromiso: botón de emergencia &apos;Pausar todos los tokens&apos; en el Admin Dashboard (accesible solo para CTO y CEO) pausa todos los contratos activos en Hedera en menos de 60 segundos. La revocación de clave sigue el procedimiento Shamir.
Post-incidente: informe en 72 horas con causa raíz, capa afectada (Hedera/Formance/ambas), acciones tomadas y medidas preventivas. El informe se adjunta al expediente del incidente en Nextcloud.
8.1.3 Suite de Regresión Dual-Ledger (US-L-36, US-L-37)
La suite de regresión valida los flujos críticos en ambas capas de forma coordinada. Un fallo en cualquier test bloquea el despliegue en mainnet hasta que se resuelva.
Test end-to-end completo: PSE confirmado → Numscript ejecutado en Formance staging → token emitido en Hedera testnet → TX ID registrado en ambos sistemas → reconciliación verifica que cuadran.
Test de distribución: Numscript distribución ejecutado → webhook al Forwarder → distribución USDC en testnet → TXs ID por holder registrados → balances Formance cuadran con Mirror Node.
Test de rollback: simular fallo en el Paso 2 del despliegue coordinado → verificar que el token HTS se pausa automáticamente en menos de 5 segundos.
Test de atomicidad de Numscript: introducir un fallo forzado en el script → verificar que todos los balances de Formance regresan al estado anterior (sin balances parciales ni estados corruptos).

1. Resumen Ejecutivo
1.1 Propósito del Documento
Este PRD define los requerimientos funcionales y el plan de trabajo de la experiencia digital del Holder/Inversor en la plataforma ACCET. Es el quinto PRD de la serie Dual-Ledger y el primero centrado en el usuario externo — el inversor que pone capital en los activos tokenizados.
El inversor es el motor del negocio: sin capital suscrito no hay activos fondados, sin activos fondados no hay fee ACCET, sin fee ACCET no hay empresa. Su experiencia digital debe ser tan buena que el inversor confíe en poner su capital en un activo que nunca ha visto físicamente, gestionado por una plataforma que puede no conocer.
La propuesta de valor central del inversor se apoya en tres pilares que este PRD debe hacer tangibles en la UI: rendimiento superior al mercado local (15% E.A. vs 9–11.5% CDT Colombia), respaldo en un activo físico verificable, y prueba on-chain independiente de cada peso invertido y cada distribución recibida.

2.2 Principios de Diseño UX para el Inversor
Confianza antes que features: el inversor que llega por primera vez necesita entender en 30 segundos qué es ACCET y por qué puede confiarle su capital. El simulador visible sin login es el primer acto de confianza.
Sin jerga blockchain para el inversor conservador: los conceptos blockchain (token, wallet, HTS, on-chain) se explican con analogías de su mundo. &apos;Tu certificado digital de inversión&apos; en lugar de &apos;token&apos;. &apos;Verificación independiente&apos; en lugar de &apos;on-chain&apos;.
Transparencia gradual: el inversor conservador no necesita ver el TX ID de Hedera para confiar. El inversor tech-savvy sí. El diseño debe ofrecer ambas capas — la simple y la verificable — sin forzar a nadie a la que no le corresponde.
Sin fricción en la conversión: cada paso del funnel (simulador → registro → KYC → suscripción → pago) debe ser el más corto posible. Si un paso puede eliminarse, se elimina. Si puede posponerse, se pospone.
Estado siempre visible: el inversor debe saber en todo momento en qué etapa está su inversión, cuándo es su próximo pago y cuánto ha ganado hasta ahora. El dashboard no es un reporte — es el estado en tiempo real de su capital.

4. Fase 1 — Descubrimiento y Conversión
4.1 Lineamientos — Fase 1
4.1.1 Simulador de Rendimiento Público (US-H-01)
El simulador es la primera pantalla que ve cualquier visitante de ACCET. No requiere login. Es el principal instrumento de conversión y debe funcionar en menos de 30 segundos desde que el usuario llega.
Slider de monto: $500 USD mínimo hasta $100,000 USD. Al mover el slider, todos los resultados se actualizan en tiempo real sin recarga de página.
Resultados mostrados simultáneamente: rendimiento a 15% E.A. semestral, distribuciones proyectadas en 4 pagos, monto total al vencimiento, todo en USD y en COP (TRM vigente del Banco República via Chainlink).
Comparativa integrada: en la misma pantalla, tres columnas — ACCET (15% E.A.) · CDT Colombia promedio (9–11.5% E.A.) · Cuenta de ahorros Colombia (3–4% E.A.). Las diferencias en pesos al vencimiento son la pieza más persuasiva de la UI.
Un solo CTA prominente desde el resultado: &apos;Ver activos disponibles&apos;. No &apos;Registrarse&apos;, no &apos;Invertir ahora&apos; — primero el marketplace, luego el registro. Reduce la fricción de conversión.
Dato financiero crítico: Los rendimientos del simulador provienen de los parámetros globales del sistema (Admin Dashboard Fase 5 US-A-38). Si el CFO cambia la tasa, el simulador se actualiza automáticamente sin despliegue de código.
4.1.2 Landing: Diferenciadores y Confianza (US-H-02)
Tabla comparativa: ACCET vs CDT vs Finca Raíz directa vs Acciones Colombia — 5 variables: rendimiento anual, monto mínimo de entrada, liquidez, complejidad operativa, transparencia del activo subyacente.
Explicación del activo subyacente en tres oraciones: &apos;Tu capital financia propietarios de inmuebles reales en Colombia. El inmueble es la garantía. Puedes verificar que existe — y verificar que recibiste cada pago — de forma independiente a través de la red Hedera.&apos;
Sección de transparencia: link al explorador Hedera (hashscan.io) con un activo piloto activo. El inversor puede ver los tokens, las distribuciones pasadas y el smart contract antes de crear su cuenta.
Logo de sandbox SFC visible en el footer. Sección &apos;Marco Legal&apos; con link al Legal Pack de muestra descargable. No se esconde la tecnología — se explica.
4.1.3 Registro Ligero con OTP (US-H-03)
Registro en dos campos: email o número WhatsApp + OTP de 6 dígitos. Sin contraseña en el primer paso — reduce abandono. La contraseña se crea después de la primera sesión.
Al completar el registro: el usuario entra en modo &apos;exploración&apos;. Puede ver todos los activos del marketplace con su información completa. No puede suscribir hasta completar KYC.

5. Fase 2 — Onboarding: KYC, AML y Wallet
5.1 Lineamientos — Fase 2
5.1.1 KYC Mobile-first con Jumio/Onfido (US-H-04)
El KYC es el primer punto de abandono del funnel. El objetivo es completarlo en menos de 5 minutos desde el celular sin que el usuario tenga que ir a ninguna oficina.
Flujo mobile-first: captura de cédula frontal + trasera + selfie con liveness detection vía Jumio/Onfido SDK. El proceso guía al usuario con instrucciones visuales en cada paso.
Soporte multipaís desde el diseño inicial: inversores colombianos (CC), venezolanos (pasaporte), US persons (SSN/EIN presentado solo como aviso, no como campo de captura en el KYC).
Resultado en &lt; 5 minutos para el caso nominal. Si el procesamiento demora, el usuario recibe una notificación push/email con ETA. No se bloquea en una pantalla de espera.
Estados claros: Aprobado (avanza al siguiente paso automáticamente) / En revisión (ETA visible) / Rechazado (razón específica + botón &apos;Corregir y reintentar&apos; con instrucciones exactas de qué mejorar).
El resultado del KYC se envía al Admin Dashboard para revisión manual si la confianza automática es baja (Admin Dashboard Fase 2, US-A-02). El usuario no percibe esta diferencia — solo ve &apos;En revisión&apos;.
5.1.2 Screening AML Silencioso (US-H-05)
El screening OFAC + SARLAFT Colombia + UN Consolidated se ejecuta en segundo plano al completar el KYC. El usuario no ve ninguna pantalla adicional ni se le pide acción.
Si el resultado es &apos;clear&apos; en menos de 30 segundos: el usuario avanza automáticamente a la pantalla de &apos;KYC Aprobado — Ya puedes invertir&apos;. Sin intervención del equipo de compliance.
Si el resultado es &apos;review&apos;: el usuario ve &apos;Estamos verificando tu perfil — recibirás un email en las próximas 2 horas&apos;. El equipo de compliance recibe la alerta en el Admin Dashboard.
Log del screening con timestamp guardado automáticamente en el expediente del inversor para auditoría regulatoria. No visible para el usuario.
5.1.3 Perfil de Riesgo y Origen de Fondos (US-H-06)
Formulario de 3 minutos con diseño de preguntas type &apos;tarjeta seleccionable&apos; (no formulario tradicional): categorías de origen de fondos (salario, ventas de activos, herencia, dividendos, otros) y perfil de riesgo (conservador / moderado / agresivo).
Cuestionario MiFID-lite en español colombiano: 5 preguntas que determinan el perfil de riesgo. Cada pregunta tiene 3 opciones presentadas como escenarios concretos, no como preguntas abstractas.
Resultado del perfil visible inmediatamente: &apos;Tu perfil es Moderado. Puedes invertir hasta $50,000 USD por activo.&apos; El inversor entiende sus límites antes de ver el marketplace.
Los datos del perfil se guardan cifrados. El inversor puede actualizarlos anualmente desde su configuración.
5.1.4 Wallet Hedera Custodial (US-H-07)
Al aprobar KYC: ACCET crea automáticamente una wallet custodial Hedera (AccountID) para el inversor. El usuario recibe una notificación: &apos;Tu cartera digital está lista. No necesitas guardar ninguna clave.&apos;
Para el inversor conservador: la wallet es invisible. Recibe sus tokens sin saber nada de Hedera. Solo ve sus inversiones en el dashboard.
Para el inversor tech-savvy: el AccountID es visible en configuración con link a hashscan.io. Opción avanzada para conectar wallet propia (HashPack / MetaMask) si lo prefiere.
La creación de la wallet se comunica al Admin Dashboard para que el equipo de Ops pueda asociar el AccountID al expediente del inversor.

El marketplace es la pantalla más importante del producto: donde el inversor decide si confía en poner su capital. Debe comunicar seguridad, transparencia y oportunidad en un solo vistazo.
Card por activo: fotografía verificada del inmueble + ciudad + tipo (residencial/comercial/NdS) + rendimiento E.A. + LTV actual + plazo + monto mínimo + barra de progreso de suscripción (&apos;68% suscrito — quedan $9,600 USD&apos;).
Badge de estado prominente: &apos;Abierto · En suscripción&apos; (verde) / &apos;Cerrando pronto — &lt; 20% disponible&apos; (amarillo) / &apos;Cerrado · Lista de espera&apos; (gris). El badge amarillo genera urgencia sin ser manipulador — es un dato real.
Filtros persistentes: tipo de activo (residencial/comercial/NdS/bodega), nivel de riesgo, jurisdicción (COL/USA), monto mínimo (slider $500–$50K USD). Badge de compatibilidad con el perfil del usuario en cada card: &apos;✓ Compatible con tu perfil moderado&apos;.
Ordenar por: rendimiento descendente, plazo más corto, fecha de apertura más reciente. Persistido en la sesión del usuario.
6.1.2 Ficha Completa del Activo con Datos On-chain (US-H-10, US-H-11, US-H-12, US-H-13)
La ficha es el equivalente al prospecto de una inversión tradicional — pero en tiempo real y verificable on-chain. Debe ser completa para el inversor institucional y comprensible para el inversor conservador.
Sección Financiera: rendimiento E.A., 4 fechas de distribución proyectadas con montos en USD y COP, yield total al vencimiento, fee ACCET (3%) desglosado, monto neto al inversor.
Sección del Inmueble: dirección exacta, m², estrato o clase (residencial/comercial), fotografías verificadas por el equipo ACCET, avalúo Mundo Hogar con fecha y monto.
Sección Legal: fideicomiso activo con número de registro, SPV Delaware (si aplica), escritura pública con número, estado del título (libre de hipotecas), certificado de libertad y tradición.
Sección On-chain: Token ID Hedera con link a hashscan.io, LTV en tiempo real desde oracle Chainlink con semáforo (verde &lt; 60% / amarillo 60–70% / rojo &gt; 70%) y timestamp del último update. Historial de distribuciones pasadas del token con TX IDs verificables.
Legal Pack descargable: botón único que descarga ZIP con escritura, certificado de libertad, avalúo y contrato fideicomiso. El hash SHA-256 del ZIP verificable en el HCS topic del activo en Hedera.
6.1.3 Comparador y Alertas (US-H-14, US-H-15)
Botón &apos;Agregar a comparador&apos; en cada card (máximo 3). La tabla comparativa muestra las mismas 8 variables para todos los activos seleccionados. El mejor valor de cada fila destacado en el color sky del branding.
Favoritos: ícono de estrella en cada activo. Lista de favoritos accesible desde el perfil. Alerta configurable cuando quede menos del 20% del cupo: &apos;Solo quedan $4,800 USD disponibles en [activo]&apos;.
6.2 Lineamientos — Fase 6.2: Suscripción y Pago
6.2.1 Simulación Final Pre-pago (US-H-16)
Antes de cualquier pago: pantalla de confirmación con monto invertido, número de tokens a recibir, precio por token, tabla de 4 pagos proyectados con fechas exactas y montos en USD y COP, desglose de fees.
Checkbox de aceptación del Contrato de Suscripción y los Términos del Token. El contrato se puede leer in-app sin salir a un PDF externo. Obligatorio antes de habilitar el botón de pago.
6.2.2 Pago PSE Colombia (US-H-17)
Lista de bancos colombianos con logos, conversión COP/USD con TRM vigente visible y editable (el inversor ve exactamente cuántos pesos va a pagar). Botón &apos;Pagar&apos; lleva al flujo PSE del banco seleccionado.
Confirmación en tiempo real al regresar del PSE. Comprobante PDF descargable inmediatamente. Si el pago falla: mensaje específico de la causa (fondos insuficientes, límite diario, etc.) con botón &apos;Reintentar&apos; sin perder el avance.
6.2.3 Pago USDC y Wire Internacional (US-H-18, US-H-19)
USDC: botón &apos;Pagar con USDC&apos; genera una dirección Hedera de destino + QR con el monto exacto. Confirmación automática en &lt; 5 segundos al detectar la TX en la red. El TX ID aparece inmediatamente en la confirmación.
Wire: instrucciones completas (beneficiary, ABA, SWIFT, referencia única) mostradas en pantalla y enviadas por email. Status tracker visible: &apos;Wire en espera → Confirmado por ACCET → Tokens emitidos&apos;.
6.2.4 Recepción de Tokens y Primera Vista del Dashboard (US-H-20, US-H-21)
Al confirmar el pago (automático para PSE/USDC, manual para wire): pantalla de &apos;Procesando tu inversión&apos; con indicador de progreso. Al recibir la señal del Forwarder via WebSocket: transición a &apos;Tokens recibidos&apos; con confetti y el TX ID de Hedera.
El dashboard aparece inmediatamente tras la emisión con los 4 KPIs superiores: Capital invertido total · Rendimiento acumulado (inicialmente $0) · Próxima distribución con fecha · Estado &apos;Activo&apos;.
Badge on-chain en cada posición: &apos;◈ Verificado en Hedera · [TX ID]&apos; con link a hashscan.io. Para el inversor conservador es opcional verlo. Para el tech-savvy es la confirmación que esperaba.

7. Fase 4 — Dashboard de Portafolio y Distribuciones
7.1 Lineamientos — Fase 7.1: Dashboard
7.1.1 Panel de Portafolio Consolidado (US-H-21, US-H-37)
El dashboard es la pantalla que el inversor visita con más frecuencia. Su trabajo principal es dar tranquilidad — confirmar que la inversión está activa, que los pagos llegaron y que el capital está seguro.
KPIs superiores (siempre visibles sin scroll): Capital invertido total USD · Rendimiento acumulado recibido USD · Próxima distribución con monto proyectado y fecha · Estado global del portafolio.
Lista de posiciones: por activo — monto invertido, número de tokens, valor actual, rendimiento recibido vs proyectado, estado (activo / vencido / en distribución). Click en cualquier posición abre la ficha completa del activo.
Badge on-chain por posición: &apos;◈ Verificado en Hedera · [TX ID de emisión]&apos;. Para el inversor avanzado: click abre hashscan.io con la TX de emisión.
Panel de analytics: gráfico de diversificación del portafolio (% por activo y por tipo), TIR global calculada en tiempo real, comparativa TIR ACCET vs CDT Colombia del mismo período.
7.1.2 Calendario de Distribuciones (US-H-23, US-H-24)
Vista calendario mensual con puntos de pago marcados por fecha. Click en un punto abre el detalle: activo, monto USD proyectado, monto COP estimado, estado (recibido/pendiente).
Vista lista alternativa: tabla con todas las distribuciones futuras ordenadas por fecha, activo, monto en USD y COP.
Exportable a Google Calendar / iCal: botón &apos;Agregar a mi calendario&apos; crea eventos con recordatorio 7 días antes para cada distribución futura.
Notificaciones configurables por canal (push/email/WhatsApp): T-7 &apos;En 7 días recibirás $750 USD de rendimiento de [activo]&apos; y T-0 &apos;Tu pago está en camino&apos;.
7.1.3 Recepción de Distribuciones (US-H-25, US-H-26, US-H-28)
Al recibir una distribución: notificación inmediata con el monto exacto y el TX ID de Hedera (para pagos USDC on-chain). El dashboard se actualiza en tiempo real con el nuevo rendimiento acumulado.
Comprobante descargable por distribución: desglose capital pendiente + intereses + retención DIAN si aplica. TX ID verificable para pagos on-chain.
Historial de distribuciones: tabla paginada filtrable por activo, año fiscal, tipo de pago. Exportable como CSV para el contador del inversor.
Reinversión automática: toggle por posición o para el portafolio completo. Notificación de confirmación 48h antes de cada reinversión con opción de cancelar.
7.1.4 Alertas de Mora y Transparencia (US-H-27)
Si el propietario de un activo tiene retraso en el pago: el inversor recibe una notificación proactiva antes de que llegue la fecha de distribución. Nunca debe enterarse buscando en el dashboard.
El panel de estado del activo muestra: días de mora del propietario, estado del período de gracia, si se activó la garantía del fideicomiso, y ETA de resolución.
Tono de la comunicación: informativo y tranquilizador, no alarmista. &apos;El propietario de [activo] tiene 3 días de retraso en su pago. La garantía del fideicomiso está activa y tu distribución está protegida. ETA: [fecha].&apos;
7.1.5 Certificado DIAN y Bóveda Digital (US-H-29, US-H-38)
Certificado de ingresos y retenciones disponible a partir del 1 de febrero del año siguiente. Valores en COP con TRM promedio del año del Banco República. Firmado digitalmente por AgoraLink S.A.S.
Bóveda digital de documentos: carpeta organizada por inversión con contrato, comprobantes de pago, certificados fiscales y resumen final. Descarga individual o ZIP completo. Búsqueda por nombre de activo, fecha o tipo de documento.

8. Fase 5 — Cierre, Analytics y Funciones Avanzadas
8.1 Lineamientos — Fase 5
8.1.1 Proceso de Vencimiento y Token Burn (US-H-31, US-H-32, US-H-33)
Notificaciones progresivas: T-90 &apos;Tu inversión en [activo] vence en 90 días. Rendimiento acumulado hasta hoy: $X USD.&apos; T-30 con opción de reinvertir en nuevos activos disponibles. T-7 con monto exacto a recibir.
Al vencimiento: pago automático del capital + último cupón. El dashboard muestra la posición como &apos;Vencida · Capital devuelto&apos; con fecha y monto exacto. Si hay retraso del propietario: comunicación proactiva con ETA + activación de garantía.
Confirmación de burn: notificación con TX ID del burn en Hedera. &apos;◈ Tus [N] tokens ACCET-NdS-001 fueron quemados · TX: [hash] · Verificar en hashscan.io&apos;. La posición pasa a &apos;Historial de inversiones completadas&apos;.
8.1.2 Resumen Financiero Final y NPS (US-H-34, US-H-35)
PDF de resumen financiero: fecha de inicio, capital invertido, 4 distribuciones recibidas con fechas y montos, capital devuelto al vencimiento, TIR final calculada. Comparativa automática: &apos;TIR ACCET vs CDT promedio Colombia del mismo período&apos;.
Código QR de verificación en el PDF: apunta al registro on-chain en Hedera que certifica todas las operaciones del activo. El inversor puede mostrar el PDF con QR a terceros como prueba de su inversión.
NPS automático post-cierre: 1 pregunta + campo de comentario opcional. Badge desbloqueado: &apos;Inversor verificado ACCET · 1 inversión completada&apos;. Sección &apos;Próximas oportunidades&apos; con 3 activos sugeridos por perfil de riesgo.
8.1.3 Mercado Secundario OTC (US-H-30)
Opción &apos;Transferir posición&apos; visible en cada activo activo. Desactivada con banner informativo hasta que el venue de mercado secundario esté habilitado (Q3 2026): &apos;Mercado secundario disponible próximamente&apos;.
Al habilitarse: la transferencia solo es posible entre inversores en la whitelist ERC-3643. Precio sugerido: valor nominal + rendimiento acumulado proporcional. OTC negociable entre las partes.
8.1.4 Cross-border y Perfiles Internacionales (US-H-40)
US persons: aviso FATCA/W-8BEN al completar el perfil. Método de pago wire USD como único disponible. Reportes en USD. No se envía certificado DIAN.
Venezuela: screening OFAC Venezuela extendido al completar el KYC. Pago exclusivamente en USDC (no PSE, no wire). Documentación fiscal adaptada a la jurisdicción del inversor.
La detección de jurisdicción se hace automáticamente por el país del documento de identidad presentado en el KYC. El inversor puede corregirla en su perfil si es diferente.
8.1.5 Seguridad, Referidos y Accesibilidad (US-H-36, US-H-39, US-H-42)
2FA obligatorio en acciones críticas: nueva inversión, retiro de capital, cambio de cuenta bancaria. Métodos: TOTP (Google Authenticator/Authy) o SMS OTP. Alerta por email al detectar login desde IP o país nuevo.
Programa de referidos: código personal único. Link listo para WhatsApp/LinkedIn/email. Panel de referidos: registrados → KYC completado → primera inversión realizada → bono acumulado. Bono: 0.5% adicional de rendimiento sobre la próxima inversión por cada referido activo.
Modo Asistido para inversores mayores de 60 años: UI simplificada a los pasos esenciales, fuente mínima 16px, contraste WCAG AA garantizado, tooltips para términos técnicos, CTA &apos;Hablar con un asesor antes de invertir&apos; prominente en la pantalla de confirmación. Flujo alternativo completo vía WhatsApp con asesor humano ACCET.

9.2 Flujos Diferenciadores del Propietario (No en el Holder)
Solicitud de tokenización: flujo de 6 pasos que va desde la carga del inmueble hasta la firma del contrato de fideicomiso. No tiene equivalente en el Holder.
Due diligence del activo: el Propietario sube los documentos legales (escritura, certificado de libertad, paz y salvo). El Holder solo los descarga en modo lectura.
Recepción del desembolso: el Propietario recibe el capital del fideicomiso una vez cerrada la suscripción. El Holder recibe las distribuciones de rendimiento.
Calendario de pagos activos (el Propietario paga): 4 cuotas semestrales al fideicomiso. El incumplimiento activa el período de gracia y la garantía. Este flujo no existe en el Holder.
Simulador de liquidez: el Propietario simula cuánto capital puede recibir por su propiedad sin venderla. La tasa que paga es 18% E.A. (no 15% que recibe el inversor — el diferencial es el fee y el margen ACCET).
9.3 Resumen del Próximo PRD
Próximo: ACCET_PRD_Propietario_v1 — Cierre de la Serie
El PRD del Propietario es el sexto y último de la serie. Cierra el ciclo completo del modelo de negocio ACCET: sin propietarios que tokenicen sus activos no hay marketplace para los inversores. El PRD debe especificar: (1) Landing y simulador de liquidez (18% E.A. — cuánto recibo por mi propiedad sin venderla); (2) Flujo de solicitud de tokenización en 6 pasos: registro → KYC → datos del inmueble → due diligence documental → avalúo Mundo Hogar → firma del contrato fideicomiso; (3) Panel de seguimiento: estado de la solicitud en el pipeline (los mismos estados del Admin Dashboard); (4) Suscripción en tiempo real: el Propietario ve cuánto de su activo fue suscrito por inversores sin ver los nombres; (5) Desembolso: recepción del capital del fideicomiso con comprobante descargable; (6) Calendario de pagos activos: 4 cuotas semestrales al fideicomiso con recordatorios y comprobante de pago; (7) Cierre: pago final del capital + confirmación del burn on-chain. La Sección 9 de este PRD (tabla de componentes reutilizables y flujos diferenciadores) es el mapa de arquitectura de UI del PRD del Propietario.

1. Resumen Ejecutivo
1.1 Propósito del Documento
Este PRD es el sexto y último de la serie Dual-Ledger ACCET. Cierra el ciclo completo del modelo de negocio: sin propietarios que tokenicen sus activos no existe el marketplace para los inversores. El Propietario es el origen de toda la cadena de valor.
La propuesta de valor para el propietario es directa: obtener liquidez sobre un activo inmobiliario sin venderlo, a una tasa de 18% E.A. — significativamente menor que un crédito hipotecario tradicional (28–35% E.A. en Colombia 2026) y sin el proceso burocrático bancario. El inmueble continúa siendo suyo. Solo paga cuatro cuotas semestrales durante dos años y recupera su título libre de gravamen.
Dado que 10 de los componentes clave están especificados en el PRD del Holder (Sección 9.1 de ese documento), este PRD se enfoca en los 5 flujos exclusivos del Propietario y en los ajustes específicos de implementación.

4. Fase 1 — Descubrimiento y Registro
4.1 Lineamientos — Fase 1
4.1.1 Landing del Propietario y Simulador de Liquidez (US-P-01, US-P-02)
La landing del propietario tiene un trabajo diferente al de la landing del inversor: no persuade sobre un rendimiento — convence de que ceder una garantía inmobiliaria temporal es más ventajoso que cualquier alternativa de financiamiento disponible.
Simulador visible sin login: campo de avalúo del inmueble (COP o USD) → calcula automáticamente el 60% LTV recibible → muestra el monto de cada cuota semestral a pagar, la tasa de 18% E.A. y el total pagado al vencimiento.
Comparativa de alternativas: ACCET (18% E.A.) vs crédito hipotecario bancario (28–35% E.A.) vs factoring (24–36% E.A.) vs venta directa. El ahorro en pesos al vencimiento es la columna más persuasiva.
FAQ específico sin jerga: &apos;¿Qué pasa si no puedo pagar una cuota?&apos; → explicación empática del período de gracia y la garantía del fideicomiso. &apos;¿Sigo siendo dueño durante los 2 años?&apos; → sí, el fideicomiso es una garantía temporal, no una venta.
Sello de confianza: logo SFC Sandbox + &apos;Fideicomiso bajo escritura pública&apos; + dato real actualizado desde el Admin Dashboard: &apos;100% de propietarios recibieron su desembolso en los últimos X días&apos;.
4.1.2 Registro Idéntico al del Holder (US-P-03)
Reutilización directa: El componente de registro OTP (H1.5) es exactamente el mismo. Solo cambia la pantalla de bienvenida: 6 pasos del propietario en lugar de 4 del inversor.
Los 6 pasos: Registro ✓ → KYC (5 min) → Datos del inmueble (10 min) → Carga de documentos (15 min) → Avalúo coordinado por ACCET (5–10 días) → Firma y desembolso (2 días).

5. Fase 2 — Onboarding y Solicitud del Activo
5.1 Lineamientos — Fase 2
5.1.1 KYC y AML del Propietario (US-P-04, US-P-05)
Reutilización directa: KYC Jumio/Onfido (H2.1–H2.4) y AML silencioso (H2.3) son idénticos. Mismo backend, mismo flujo, misma integración con el Admin Dashboard.
Al aprobar KYC del propietario: el Admin Dashboard crea el expediente en el pipeline de solicitudes (US-A-01). El propietario avanza automáticamente al formulario del inmueble.
LAFT adicional: verificación de que el inmueble no aparece en bases de datos de activos en litigio o con embargos vigentes (Registro de Instrumentos Públicos).
5.1.2 Formulario del Inmueble (US-P-06, US-P-07)
El formulario del inmueble debe ser lo suficientemente completo para el equipo de Ops, pero simple para que el propietario lo complete en 10 minutos desde su celular.
Paso 1 — Datos básicos: tipo de inmueble (residencial/comercial/bodega/NdS), dirección con autocompletado Google Maps, área en m², estrato o clase, año de construcción, estado de ocupación actual.
Paso 2 — Datos financieros: avalúo estimado en COP, existencia de hipoteca bancaria activa (sí/no). Si sí: banco, saldo actual y cuota mensual. La hipoteca activa no bloquea la solicitud, pero informa el análisis de LTV.
Paso 3 — Monto solicitado: slider sobre el avalúo estimado con tope de 60% LTV calculado en tiempo real. El propietario puede solicitar menos del máximo para una cuota más baja.
Guardado automático cada 30 segundos. El propietario puede retomar desde cualquier dispositivo.
5.1.3 Carga Documental (US-P-08, US-P-09)
La carga de documentos es donde más propietarios abandonan el funnel. El diseño debe minimizar la fricción con instrucciones visuales claras y validación inmediata.
5 documentos con instrucciones visuales: escritura pública + certificado de libertad y tradición (max 30 días de vigencia) + paz y salvo predial + documento de identidad + reglamento PH si aplica.
Cada documento tiene un card con: instrucción breve, captura por cámara mobile o PDF, preview inmediato, estado (pendiente/cargado/vencido/rechazado). Validación de calidad de imagen antes de enviar.
Al completar la carga: la solicitud pasa a estado &apos;Due Diligence&apos; en el Admin Dashboard. El propietario recibe: &apos;Documentos recibidos — revisión en los próximos 2 días hábiles.&apos;
5.1.4 Seguimiento del Estado de la Solicitud (US-P-10, US-P-11)
Panel de estado con barra de progreso de los 6 pasos. El propietario sabe exactamente en qué paso está su solicitud en todo momento sin necesidad de contactar al equipo.
Notificación automática en cada transición de estado. Si se rechaza un documento: notificación con el documento específico, la razón y botón &apos;Reemplazar documento&apos;.
Al solicitar el avalúo: el propietario ve el estado del avalúo en su dashboard (Solicitado → Visita programada → En proceso → Resultado disponible). Al recibir el avalúo: recalcula el monto disponible si difiere del estimado y notifica al propietario.

6. Fase 3 — Firma, Emisión y Desembolso
6.1 Lineamientos — Fase 6.1: Contrato y Firma Digital
6.1.1 Revisión y Firma del Contrato de Fideicomiso (US-P-12, US-P-13, US-P-14)
La firma del contrato es el momento de mayor importancia jurídica. La UX debe hacer que el propietario entienda exactamente lo que firma sin minimizar los detalles ni abrumarlo con legalese.
Visor de contrato in-app con resaltado de cláusulas clave: monto, tasa, plazo, fechas de pago, condiciones de mora, proceso de ejecución de la garantía. Glosario de términos legales accesible tocando cualquier palabra subrayada.
4 checkboxes individuales obligatorios antes de firmar (no un único &apos;acepto todo&apos;): monto recibido confirmado · tasa 18% E.A. confirmada · fechas de las 4 cuotas confirmadas · consecuencias del incumplimiento leídas.
Firma digital con OTP de 6 dígitos + sello de tiempo AgoraLink S.A.S. El PDF firmado queda en la bóveda digital del propietario y en el expediente del Admin Dashboard.
Hash SHA-256 del contrato registrado en el HCS topic del activo en Hedera al momento de la firma. Verificable en hashscan.io en cualquier momento.
6.2 Lineamientos — Fase 6.2: Suscripción y Desembolso
6.2.1 Panel de Suscripción (US-P-15, US-P-16) y Desembolso (US-P-17, US-P-18)
Post-emisión: panel de suscripción con porcentaje del activo suscrito, capital recaudado en USD y número de inversores activos — sin mostrar nombres ni montos individuales de los inversores.
Al alcanzar el 100% de suscripción: notificación automática con ETA del desembolso (3–5 días hábiles).
Al ejecutar el desembolso desde el Admin Dashboard: notificación al propietario con monto exacto y referencia bancaria. Comprobante PDF descargable: monto bruto, fee ACCET (3%), gastos fideicomiso, monto neto recibido.
NPS inmediato 7 días después del desembolso: &apos;¿Recomendarías ACCET a otro propietario?&apos;

7. Fase 4 — Gestión de Pagos Activos al Fideicomiso
7.1 Lineamientos — Fase 4
7.1.1 Calendario de Pagos y Ejecución (US-P-23, US-P-24, US-P-25, US-P-26)
El calendario de pagos es la pantalla que el propietario visita más durante los 2 años del contrato. Su trabajo es ser un recordatorio claro y un facilitador del pago.
Vista calendario con las 4 fechas de pago al fideicomiso. Cada fecha muestra el monto exacto (capital + intereses del semestre), estado (próximo/pagado/vencido) y botón de pago.
Recordatorios automáticos: T-15, T-7 y T-0 de cada fecha de vencimiento. Canal configurable por el propietario.
Pago desde la app vía PSE (propietarios colombianos) o transferencia bancaria con instrucciones. Al confirmar PSE: notificación automática al equipo de Ops en el Admin Dashboard para su validación. El propietario ve &apos;Pago enviado — en verificación&apos;.
Al confirmar el equipo: el propietario recibe &apos;✓ Pago confirmado&apos; y el sistema dispara el Numscript de distribución en Formance → Forwarder → distribuciones a los holders.
7.1.2 Período de Gracia y Mora (US-P-27, US-P-28)
Si el propietario no paga en la fecha de vencimiento: período de gracia automático de 15 días. Comunicación empática: &apos;Tu cuota venció hace X días. Tienes hasta [fecha] para pagar sin consecuencias adicionales.&apos;
Semáforo de estado del contrato visible para el propietario: verde (al día) / amarillo (en período de gracia) / rojo (garantía del fideicomiso activa).
Al día 15 de mora: comunicación de activación de garantía del fideicomiso. Tono empático pero claro. El equipo legal de ACCET se comunica para coordinar el plan de pagos.
7.1.3 Salida Anticipada (US-P-29, US-P-30)
El propietario puede solicitar cancelación anticipada desde la app. El sistema calcula el saldo pendiente (capital + intereses prorrateados + penalidad si aplica) antes de confirmar.
Al confirmar: aprobación del CFO desde el Admin Dashboard → pago al fideicomiso → burn de tokens → liberación de la garantía.

8. Fase 5 — Cierre, Cancelación y Nuevo Ciclo
8.1 Lineamientos — Fase 5
8.1.1 Cierre del Contrato (US-P-31, US-P-32, US-P-33)
Notificaciones T-90/30/7 de vencimiento con capital pendiente y ETA de liberación del activo.
Pantalla de cuota final con desglose del último pago: capital original + último semestre de intereses. El propietario entiende exactamente qué está pagando.
Post-pago final: el Admin Dashboard ejecuta la distribución final a los holders y el burn de tokens. El propietario recibe: &apos;¡Tu contrato ACCET ha sido cancelado! Los tokens de tu inmueble fueron destruidos en Hedera. Tu propiedad está libre.&apos;
TX ID del burn visible en el dashboard del propietario con link a hashscan.io.
8.1.2 Certificado de Cancelación y Resumen (US-P-34, US-P-35)
PDF de certificado de cancelación: monto total pagado, desglose por cuota, capital recuperado por el fideicomiso, fecha de liberación de la garantía, TX ID del burn en Hedera. Código QR que enlaza al HCS topic del activo para verificación criptográfica ante terceros (banco, notaría, comprador eventual).
Resumen financiero del ciclo: capital recibido, 4 cuotas pagadas, total de intereses pagados, costo efectivo (TEA), comparativa &apos;vs crédito bancario equivalente&apos; con el ahorro real en pesos.
8.1.3 NPS, Referidos y Re-solicitud (US-P-36, US-P-37)
NPS post-cierre + badge &apos;Propietario verificado ACCET · 1 ciclo completado&apos;. Programa de referidos del propietario: 0.25% de descuento en la tasa del siguiente contrato por cada propietario que complete un ciclo referido.
Flujo de re-solicitud: el activo está disponible para un nuevo ciclo con el formulario pre-llenado con los datos del ciclo anterior. El proceso de re-solicitud es significativamente más rápido: KYC ya aprobado, activo ya conocido por el equipo de Ops.

9.2 Secuencia de Implementación Recomendada
Semanas 1–3: Formance PRD Fase 1 + Hedera PRD Fase 1 + Forwarder PRD Fase 1 en paralelo. Establecer las conexiones base.
Semanas 4–6: Admin Dashboard Fase 1 (auth + RBAC + audit trail). Holder Fase 1 + Propietario Fase 1 trabajando contra mocks del backend.
Semanas 7–10: Hedera PRD Fase 2 (contratos + auditoría) + Admin Dashboard Fase 2 (pipeline) + Holder Fase 2 + Propietario Fase 2 (KYC + onboarding).
Semanas 11–15: Forwarder PRD Fases 2–3 + Hedera PRD Fase 3 + Admin Dashboard Fase 3 + Holder Fase 3 + Propietario Fase 3. Primera emisión en mainnet.
Semanas 16–20: Admin Dashboard Fase 4 + Holder Fase 4 + Propietario Fase 4. Primera distribución semestral automática completa — el hito de validación más importante del sistema.
Semanas 21–25: Fases 5 de todos los PRDs en paralelo. Seguridad, DR, reportes regulatorios, cierre completo del primer activo.

1. Resumen Ejecutivo
1.1 Proposito del Documento - Cierre de la Serie
Este PRD define los requerimientos funcionales y el plan de trabajo de la experiencia digital del Propietario en la plataforma ACCET. Es el sexto y ultimo PRD de la serie Dual-Ledger, y cierra el ciclo completo del modelo de negocio: el Propietario es el origen de todos los activos del marketplace. Sin propietarios que tokenicen sus inmuebles no hay activos, sin activos no hay inversores, sin inversores no hay AUM, sin AUM no hay empresa.
La experiencia del Propietario es radicalmente diferente a la del Inversor: el Propietario no viene a ganar rendimientos, viene a resolver un problema de liquidez. Su inmueble vale dinero pero ese dinero esta congelado en ladrillo. ACCET le ofrece liquidar hasta el 60% del avaluo sin vender ni perder la propiedad. La experiencia digital debe hacer ese proceso mas simple, transparente y confiable que ir a un banco.

2.2 Flujos Exclusivos del Propietario
Simulador de liquidez: cuanto capital puede recibir por su inmueble (60% LTV del avaluo). La tasa que paga es 18% E.A. La diferencia con el 15% del inversor es el margen operativo de ACCET.
Solicitud de tokenizacion en 6 pasos: flujo guiado desde los datos basicos del inmueble hasta la firma digital del contrato de fideicomiso. El flujo mas diferenciador del producto.
Seguimiento del due diligence y avaluo: el propietario ve en tiempo real el estado de su solicitud dentro del pipeline del Admin Dashboard, sin ver el panel interno.
Desembolso y suscripcion en tiempo real: el propietario recibe el capital del fideicomiso una vez cerrada la suscripcion y puede ver cuanto de su activo fue fondado por inversores.
Pagos activos al fideicomiso: 4 cuotas semestrales de interes mas el capital al vencimiento. Pago desde la app con PSE/USDC/transferencia y comprobante inmediato.

4. Fase 1 - Descubrimiento y Registro
4.1 Lineamientos
4.1.1 Landing y Simulador de Liquidez (US-P-01, US-P-02)
La landing del Propietario transmite un mensaje central diferente al del Inversor: no es gana mas rendimiento, es libera el valor de tu propiedad sin venderla. El simulador es el primer argumento y debe funcionar en 30 segundos.
Simulador visible sin login: campo de avaluo estimado del inmueble. Al ingresar el monto: calcula automaticamente 60% LTV, muestra el monto disponible en USD y COP (TRM Chainlink vigente), y la cuota semestral al 18% E.A.
Ejemplo ancla siempre visible: avaluo $100,000 USD, monto disponible $60,000 USD, cuota semestral $5,400 USD. El propietario entiende el modelo completo en 30 segundos sin leer texto adicional.
Comparativa banco vs ACCET en la misma pantalla: tiempo de desembolso, LTV, tipo de garantia (hipoteca vs fideicomiso), costo total de la operacion. La diferencia mas persuasiva es el tipo de garantia.
FAQ en espanol colombiano sin jerga blockchain: que es un fideicomiso (con analogia simple), que pasa si no pago (periodo de gracia explicado), puedo salir antes (condiciones de salida anticipada con el 2.5% de fee).
Link a un activo real tokenizado en hashscan.io: el propietario puede ver que otros inmuebles ya estan registrados en la red antes de crear su cuenta. Acto de transparencia que genera confianza.
4.1.2 Registro con Soporte para Persona Juridica (US-P-03, US-P-06)
Registro base identico al del Inversor: email o WhatsApp + OTP de 6 digitos. Mismo componente sin modificacion.
Selector de tipo de propietario al crear la cuenta: Persona Natural (CC) o Persona Juridica (NIT). Si es Persona Juridica, el KYC se amplia con: NIT, Camara de Comercio vigente, datos del representante legal con poderes suficientes.
Pantalla de bienvenida con mapa de 6 pasos especifico del propietario: Registro hecho / KYC (5 min) / Solicitud del activo (15 min) / Due diligence ACCET (3-10 dias) / Avaluo Mundo Hogar (5-10 dias) / Firma y desembolso (2-5 dias).
El codigo del asesor inmobiliario aliado se asocia automaticamente al registro si el propietario viene referido. El asesor recibe una notificacion y puede seguir el avance de su referido.
Personas juridicas: El KYC de persona juridica requiere revision manual del equipo de Compliance antes de habilitar la solicitud. Tiempo estimado: 24 horas habiles adicionales vs la aprobacion automatica de persona natural.

5. Fase 2 - Solicitud del Activo y Due Diligence
5.1 Lineamientos
5.1.1 Formulario de Solicitud en 6 Pasos (US-P-08, US-P-09)
El formulario es el producto diferenciador. Un banco exigiria visitas fisicas, multiples firmas y semanas de espera. ACCET lo resuelve en una sesion de 15 minutos desde el celular con guardado de progreso entre sesiones.
Paso 1 - Datos del inmueble: tipo (residencial/comercial/bodega/lote), direccion exacta con autocompletado via Google Places API, metros cuadrados, estrato o uso del suelo, ano de construccion.
Paso 2 - Documentos basicos: escritura publica (PDF) + certificado de libertad y tradicion vigente (menos de 30 dias). El sistema verifica que el numero de matricula inmobiliaria del certificado coincide con el de la escritura.
Paso 3 - Situacion del inmueble: hay hipoteca activa, hay otro gravamen, esta arrendado. Las respuestas determinan la elegibilidad y las condiciones del contrato de fideicomiso.
Paso 4 - Datos bancarios: cuenta bancaria de destino del desembolso con verificacion de titularidad. Para USDC: wallet Hedera del propietario o nueva wallet custodial creada por ACCET.
Paso 5 - Monto solicitado: slider entre el minimo ($5,000 USD) y el maximo (60% del avaluo estimado). El propietario ve en tiempo real la cuota semestral resultante al 18% E.A.
Paso 6 - Confirmacion y envio: resumen de todos los datos + checkbox de declaracion de veracidad + envio. Al enviar: numero de caso unico generado e email de confirmacion enviado.
5.1.2 Seguimiento del Due Diligence (US-P-10, US-P-11)
Stepper visual en el dashboard del propietario: Solicitud enviada / Revision de documentos / Avaluo en proceso / Aprobacion del comite / Firma de contratos / Desembolso. El propietario sabe en que paso esta y que sigue.
Solicitud de documentos adicionales: si el equipo de Ops necesita un documento, el propietario recibe una notificacion especifica y puede subirlo desde el dashboard sin llamar a nadie.
ETA visible en cada paso. Si el paso tarda mas de lo estimado: notificacion proactiva al propietario con la razon del retraso y el nuevo ETA. Nunca el propietario descubre el retraso llamando.
Resultado del avaluo Mundo Hogar: notificacion al propietario con tres opciones: Aceptar el monto / Solicitar reconsideracion con justificacion / Desistir sin costo. 5 dias habiles para responder.
Desglose completo de costos antes de la firma: fee de originacion ACCET (2%), costos de formalizacion del fideicomiso, total intereses en 2 anos (4 pagos x cuota semestral), costo efectivo total de la operacion. Sin letra pequena.
5.1.3 Firma Digital del Contrato (US-P-13)
Firma digital del contrato de colateralizacion y del acuerdo de fideicomiso desde la app. El propietario lee el contrato in-app sin salir a un PDF externo, y firma con clave electronica o con firma OTP.
Hash del contrato firmado registrado en Hedera HCS para inmutabilidad permanente. El propietario recibe el contrato PDF firmado en su email y en su boveda digital.

6. Fase 3 - Emision del Token y Desembolso
6.1 Lineamientos
6.1.1 Confirmacion On-chain del Token (US-P-16, US-P-20)
Cuando el CTO despliega el contrato ERC-3643 en Hedera y el Forwarder emite los tokens, el propietario ve por primera vez que su inmueble existe como activo digital verificable en la blockchain. Es el momento emocionalmente mas importante del proceso.
Notificacion: Tu activo ACCET-NdS-001 ha sido tokenizado en Hedera Hashgraph. Token ID: [X]. Puedes verificarlo en hashscan.io.
Pantalla de confirmacion on-chain: Token ID, Token Name, estandar (ERC-3643 / Hedera HTS), red (Hedera mainnet), direccion del smart contract, supply emitido, link a hashscan.io.
Para el propietario no tecnico: explicacion en lenguaje simple - Este codigo es el certificado digital de tu propiedad en la red Hedera. No puedes perderlo, no puede ser alterado y cualquier persona puede verificarlo.
6.1.2 Suscripcion en Tiempo Real (US-P-18)
Barra de progreso de suscripcion en el dashboard: 68% suscrito - $40,800 de $60,000 USD fondados. Actualizada en tiempo real via WebSocket. El propietario no ve los nombres ni montos individuales de los inversores.
Al llegar al 100%: notificacion inmediata con el monto total fondado y la fecha estimada del desembolso.
6.1.3 Desembolso del Capital (US-P-17, US-P-19, US-P-21)
Estado del desembolso en tiempo real: Fondos transferidos desde el fideicomiso / En transito / Acreditado en tu cuenta. El propietario no necesita llamar al banco para verificar.
Para propietarios que prefieren USDC: desembolso a su wallet Hedera con conversion al precio de mercado (TRM Chainlink) y TX ID verificable on-chain.
Certificado de la operacion descargable: monto desembolsado neto (capital bruto menos fee originacion 2% menos costos fideicomiso), fecha, referencia de transferencia o TX ID Hedera.

7. Fase 4 - Gestion del Contrato Activo
7.1 Lineamientos
7.1.1 Calendario de Pagos Activos (US-P-23)
El Propietario paga cuotas semestrales de interes al fideicomiso durante 2 anos. Este calendario es el opuesto del de distribuciones del Inversor: misma UI, flujo contrario.
Vista de calendario mensual: las 4 fechas de pago marcadas. Click en cada fecha: monto total a pagar, interes semestral al 18% E.A., intereses en mora si aplica.
Contador de dias hasta el siguiente pago con el monto exacto. Notificaciones T-30/7/1 configurables por canal.
Exportable a Google Calendar / iCal: mismo componente del Holder con las 4 fechas de pago del propietario.
7.1.2 Pago desde la App (US-P-24)
PSE Colombia: identico al flujo del Holder PRD - lista bancos + conversion COP/USD TRM Chainlink + confirmacion tiempo real + comprobante PDF.
USDC desde wallet Hedera: el Forwarder confirma el pago en menos de 5 segundos y registra la cuota como pagada en Formance. TX ID visible en el comprobante.
Transferencia bancaria manual: el propietario recibe datos de la cuenta del fideicomiso con su referencia unica y sube el comprobante en la app para confirmacion del equipo de Ops.
7.1.3 Oracle LTV, Periodo de Gracia y Salida Anticipada (US-P-25, US-P-26, US-P-30)
Panel estado del colateral: valor actual del inmueble segun oracle Chainlink, LTV actual, semaforo verde/amarillo/rojo. Si LTV supera 65%: notificacion informativa. Si supera 70%: notificacion urgente con acciones posibles.
Periodo de gracia in-app: el propietario puede solicitar hasta 30 dias adicionales. La pantalla muestra la tasa de mora aplicable por los dias de retraso antes de confirmar.
Salida anticipada: monto total a pagar (capital pendiente + fee 2.5%) calculado en tiempo real. Al confirmar: el Forwarder recibe la instruccion de burn, el fideicomiso libera la propiedad y se genera el certificado de cancelacion.

1. Resumen Ejecutivo
1.1 Propósito del Documento
Este PRD es el sexto y último de la serie Dual-Ledger ACCET. Cierra el ciclo completo del modelo de negocio: sin propietarios que tokenicen sus activos no existe el marketplace para los inversores. El Propietario es el origen de toda la cadena de valor.
La propuesta de valor para el propietario es directa: obtener liquidez sobre un activo inmobiliario sin venderlo, a una tasa de 18% E.A. — significativamente menor que un crédito hipotecario tradicional (28–35% E.A. en Colombia 2026) y sin el proceso burocrático bancario. El inmueble continúa siendo suyo. Solo paga cuatro cuotas semestrales durante dos años y recupera su título libre de gravamen.
Dado que 10 de los componentes clave están especificados en el PRD del Holder (Sección 9.1 de ese documento), este PRD se enfoca en los 5 flujos exclusivos del Propietario y en los ajustes específicos de implementación.

Hash SHA-256 del contrato registrado en el HCS topic del activo en Hedera al momento de la firma. Verificable en hashscan.io en cualquier momento.
6.2 Lineamientos — Fase 6.2: Suscripción y Desembolso
6.2.1 Panel de Suscripción (US-P-15, US-P-16) y Desembolso (US-P-17, US-P-18)
Post-emisión: panel de suscripción con porcentaje del activo suscrito, capital recaudado en USD y número de inversores activos — sin mostrar nombres ni montos individuales de los inversores.
Al alcanzar el 100% de suscripción: notificación automática con ETA del desembolso (3–5 días hábiles).
Al ejecutar el desembolso desde el Admin Dashboard: notificación al propietario con monto exacto y referencia bancaria. Comprobante PDF descargable: monto bruto, fee ACCET (3%), gastos fideicomiso, monto neto recibido.
NPS inmediato 7 días después del desembolso: &apos;¿Recomendarías ACCET a otro propietario?&apos;

7. Fase 4 — Gestión de Pagos Activos al Fideicomiso
7.1 Lineamientos — Fase 4
7.1.1 Calendario de Pagos y Ejecución (US-P-23, US-P-24, US-P-25, US-P-26)
El calendario de pagos es la pantalla que el propietario visita más durante los 2 años del contrato. Su trabajo es ser un recordatorio claro y un facilitador del pago.
Vista calendario con las 4 fechas de pago al fideicomiso. Cada fecha muestra el monto exacto (capital + intereses del semestre), estado (próximo/pagado/vencido) y botón de pago.
Recordatorios automáticos: T-15, T-7 y T-0 de cada fecha de vencimiento. Canal configurable por el propietario.
Pago desde la app vía PSE (propietarios colombianos) o transferencia bancaria con instrucciones. Al confirmar PSE: notificación automática al equipo de Ops en el Admin Dashboard para su validación. El propietario ve &apos;Pago enviado — en verificación&apos;.
Al confirmar el equipo: el propietario recibe &apos;✓ Pago confirmado&apos; y el sistema dispara el Numscript de distribución en Formance → Forwarder → distribuciones a los holders.
7.1.2 Período de Gracia y Mora (US-P-27, US-P-28)
Si el propietario no paga en la fecha de vencimiento: período de gracia automático de 15 días. Comunicación empática: &apos;Tu cuota venció hace X días. Tienes hasta [fecha] para pagar sin consecuencias adicionales.&apos;
Semáforo de estado del contrato visible para el propietario: verde (al día) / amarillo (en período de gracia) / rojo (garantía del fideicomiso activa).
Al día 15 de mora: comunicación de activación de garantía del fideicomiso. Tono empático pero claro. El equipo legal de ACCET se comunica para coordinar el plan de pagos.
7.1.3 Salida Anticipada (US-P-29, US-P-30)
El propietario puede solicitar cancelación anticipada desde la app. El sistema calcula el saldo pendiente (capital + intereses prorrateados + penalidad si aplica) antes de confirmar.
Al confirmar: aprobación del CFO desde el Admin Dashboard → pago al fideicomiso → burn de tokens → liberación de la garantía.

8. Fase 5 — Cierre, Cancelación y Nuevo Ciclo
8.1 Lineamientos — Fase 5
8.1.1 Cierre del Contrato (US-P-31, US-P-32, US-P-33)
Notificaciones T-90/30/7 de vencimiento con capital pendiente y ETA de liberación del activo.
Pantalla de cuota final con desglose del último pago: capital original + último semestre de intereses. El propietario entiende exactamente qué está pagando.
Post-pago final: el Admin Dashboard ejecuta la distribución final a los holders y el burn de tokens. El propietario recibe: &apos;¡Tu contrato ACCET ha sido cancelado! Los tokens de tu inmueble fueron destruidos en Hedera. Tu propiedad está libre.&apos;
TX ID del burn visible en el dashboard del propietario con link a hashscan.io.
8.1.2 Certificado de Cancelación y Resumen (US-P-34, US-P-35)
PDF de certificado de cancelación: monto total pagado, desglose por cuota, capital recuperado por el fideicomiso, fecha de liberación de la garantía, TX ID del burn en Hedera. Código QR que enlaza al HCS topic del activo para verificación criptográfica ante terceros (banco, notaría, comprador eventual).
Resumen financiero del ciclo: capital recibido, 4 cuotas pagadas, total de intereses pagados, costo efectivo (TEA), comparativa &apos;vs crédito bancario equivalente&apos; con el ahorro real en pesos.
8.1.3 NPS, Referidos y Re-solicitud (US-P-36, US-P-37)
NPS post-cierre + badge &apos;Propietario verificado ACCET · 1 ciclo completado&apos;. Programa de referidos del propietario: 0.25% de descuento en la tasa del siguiente contrato por cada propietario que complete un ciclo referido.
Flujo de re-solicitud: el activo está disponible para un nuevo ciclo con el formulario pre-llenado con los datos del ciclo anterior. El proceso de re-solicitud es significativamente más rápido: KYC ya aprobado, activo ya conocido por el equipo de Ops.

9.2 Secuencia de Implementación Recomendada
Semanas 1–3: Formance PRD Fase 1 + Hedera PRD Fase 1 + Forwarder PRD Fase 1 en paralelo. Establecer las conexiones base.
Semanas 4–6: Admin Dashboard Fase 1 (auth + RBAC + audit trail). Holder Fase 1 + Propietario Fase 1 trabajando contra mocks del backend.
Semanas 7–10: Hedera PRD Fase 2 (contratos + auditoría) + Admin Dashboard Fase 2 (pipeline) + Holder Fase 2 + Propietario Fase 2 (KYC + onboarding).
Semanas 11–15: Forwarder PRD Fases 2–3 + Hedera PRD Fase 3 + Admin Dashboard Fase 3 + Holder Fase 3 + Propietario Fase 3. Primera emisión en mainnet.
Semanas 16–20: Admin Dashboard Fase 4 + Holder Fase 4 + Propietario Fase 4. Primera distribución semestral automática completa — el hito de validación más importante del sistema.
Semanas 21–25: Fases 5 de todos los PRDs en paralelo. Seguridad, DR, reportes regulatorios, cierre completo del primer activo.
